# braemon/.doc.md

**TL;DR:** Braemon is the React 18 SPA frontend for the multi-agent memory chatbot.
It wraps Chainlit's WebSocket chat UI with a custom multi-page workspace (chats,
documents, memories, diagnostics), a dark-only "Stitch" design system, and
auto-login auth. All backend interaction goes through Vite-proxied `/api/*` REST
calls and Chainlit's `@chainlit/react-client` WebSocket hooks.

---

## Architecture overview

Braemon is a single-page application built with React Router v7, Recoil
state management, and Tailwind CSS v4. It does **not** use TypeScript — all
components are plain JSX.

```text
src/
├── main.jsx          — bootstrap: RecoilRoot, ChainlitAPI, AuthWrapper, ErrorBoundary
├── App.jsx           — route definitions (6 routes)
├── api.js            — REST client (18 functions for /api/* endpoints)
├── index.css         — Stitch design tokens + custom classes
├── components/
│   ├── Layout.jsx        — top nav bar + <Outlet/>
│   ├── ChainlitChat.jsx  — core chat experience via Chainlit SDK hooks
│   ├── ChatActions.jsx   — inline end/reactivate/consolidate/delete buttons
│   └── ErrorBoundary.jsx — React class-component crash boundary
└── pages/
    ├── Home.jsx       — landing: new chat + quick nav cards + recent threads
    ├── Chat.jsx       — sidebar (active/inactive threads) + ChainlitChat
    ├── Chats.jsx      — full thread browser (active + inactive tables)
    ├── Documents.jsx  — document library (active + inactive, search, upload)
    ├── Memories.jsx   — expandable memory vault (active + inactive)
    └── Diagnostics.jsx — system stats + build info + environment
```

### Component tree (render order)

```text
main.jsx
  <RecoilRoot>
    <ChainlitContext.Provider>
      <ErrorBoundary>
        <AuthWrapper>
          <BrowserRouter>
            <Layout>              ← fixed top nav, glass-panel body
              <Home />           ← /
              <Chat />           ← /chat/:chatId?
                → <ChainlitChat />
                → <ChatActions />
              <Chats />          ← /chats
                → <ChatActions />
              <Documents />      ← /documents
              <Memories />       ← /memories
              <Diagnostics />    ← /diagnostics
```

### Data flow

All data flows through two channels:

1. **REST API** (`src/api.js`): CRUD for chats, documents, memories, and stats.
   All calls go to `/api/*`, proxied by Vite to `localhost:8000`. Responses are
   plain JSON.

2. **Chainlit WebSocket** (`@chainlit/react-client`): Real-time chat messages,
   file uploads, session lifecycle. The SDK manages the Socket.IO connection,
   auth cookie forwarding, and streamed assistant responses.

The two channels are independent: REST manages entity lifecycle (create,
end, fork, activate, deactivate, delete), while WebSocket handles the live
chat turn (send message, receive streaming response, upload files).

## Routing and pages

React Router v7 with `<BrowserRouter>`. All routes are wrapped in `<Layout>`.

| Route | Page | Description |
|---|---|---|
| `/` | Home | Landing page. Shows recent chats, quick-nav cards to Chats/Documents/Memories, and a prominent New Chat button. |
| `/chat/:chatId?` | Chat | Two-panel layout: thread sidebar (active/inactive) + `<ChainlitChat>`. If no `chatId`, creates a new chat on connect. |
| `/chats` | Chats | Full thread browser. Active threads at top, inactive below. Each row shows title, date, and hover-revealed `<ChatActions>`. |
| `/documents` | Documents | Document library. Active and inactive sections. Search bar, quick stats bar (counts), per-document deactivate/activate/delete. |
| `/memories` | Memories | Memory vault. Expandable row-per-memory with full content in collapsed detail. Active/inactive sections with archive/delete. |
| `/diagnostics` | Diagnostics | Read-only system panel. Shows API stats (version, counts), build info, and browser environment (user agent, screen size, online status). |

## Chainlit SDK integration

Braemon depends on `@chainlit/react-client` v0.4.2 for the core chat
experience. The SDK provides:

### Client setup (`main.jsx`)

```jsx
const apiClient = new ChainlitAPI(window.location.origin, "webapp");
// Wrapped in <ChainlitContext.Provider value={apiClient}>
```

The `ChainlitAPI` instance handles:

- WebSocket (Socket.IO) connection to the Chainlit backend
- Session ID management via Recoil `sessionIdState`
- Auth cookie/session propagation

### Hooks used in `ChainlitChat.jsx`

| Hook | Purpose |
|---|---|
| `useChatSession()` | `connect()`, `disconnect()`, `idToResume` — explicit WebSocket lifecycle. UUID session per mount via `sessionIdState`. |
| `useChatMessages()` | `messages` — reactive array of all chat messages (assistant steps, tool calls, user messages). |
| `useChatInteract()` | `sendMessage()`, `uploadFile()`, `setIdToResume` — send user text/files, resume existing chats. |
| `useChatData()` | `loading`, `connected` — connection and generation state flags. |

### Connection lifecycle

1. Component mounts with `chatId` from URL params.
2. `sessionIdState` is set to a fresh UUID (one per mount).
3. If `chatId` is present, `setIdToResume(chatId)` tells Chainlit to resume history.
4. `connect({ userEnv: {} })` opens the WebSocket.
5. On unmount, `disconnect()` cleans up.

### File upload

Uses `uploadFile()` from Chainlit SDK. Returns a `{ promise, ... }` pair — the
promise resolves to a `fileRef` object passed to `sendMessage()`. Progress
(fraction 0–1) is surfaced via the `onProgress` callback and displayed in the
UI as a percentage bar. The file input is reset after selection so the same
file can be re-selected.

### Message flattening

Chainlit nests tool-call steps inside messages under `msg.steps[]`. Braemon
flattens these recursively — any step with `output` or `elements` becomes a
visible message. This ensures tool responses and sub-steps appear inline.

### Processing stage indicator

While `loading` is true, a spinner shows either "Indexing files" or "Generating
answer". The stage is controlled by `window.postMessage` events from the
backend (command `"processing-stage"`, source `"memory-chatbot-ui"`). A passive
`message` event listener in the component updates the label in real time.

## REST API layer

`src/api.js` exports 18 async functions, all targeting `/api/*` on the same
origin. In development, Vite proxies these to `localhost:8000`.

### Chat endpoints

| Function | Method | Path | Notes |
|---|---|---|---|
| `fetchChats({ cursor, search })` | GET | `/api/chats` | Optional pagination and search |
| `fetchChatMessages(chatId)` | GET | `/api/chats/:id/messages` | Returns message history |
| `createChat()` | POST | `/api/chats` | Returns `{ chat_id }` |
| `forkChat(chatId)` | POST | `/api/chats/:id/fork` | Creates a fork; returns `{ chat_id }` |
| `endChat(chatId)` | POST | `/api/chats/:id/end` | Marks chat inactive |
| `reactivateChat(chatId)` | POST | `/api/chats/:id/reactivate` | Re-activates a chat |
| `deleteChat(chatId)` | DELETE | `/api/chats/:id` | Hard delete |
| `consolidateChat(chatId)` | POST | `/api/chats/:id/consolidate` | Triggers memory consolidation |

### Document endpoints

| Function | Method | Path | Notes |
|---|---|---|---|
| `fetchDocuments({ status })` | GET | `/api/documents` | Optional status filter |
| `uploadDocumentFile(file, onProgress)` | POST | `/api/documents/upload` | **XHR** with progress events; FormData |
| `deleteDocument(docId)` | DELETE | `/api/documents/:id` | Hard delete |
| `deactivateDocument(docId)` | POST | `/api/documents/:id/deactivate` | Soft-delete (status→"deleted") |
| `activateDocument(docId)` | POST | `/api/documents/:id/activate` | Restore (status→"Ready") |

### Memory endpoints

| Function | Method | Path | Notes |
|---|---|---|---|
| `fetchMemories({ status })` | GET | `/api/memories` | Optional status filter |
| `deactivateMemory(memoryId)` | POST | `/api/memories/:id/deactivate` | Soft-delete |
| `activateMemory(memoryId)` | POST | `/api/memories/:id/activate` | Restore |
| `deleteMemory(memoryId)` | DELETE | `/api/memories/:id` | Hard delete |

### Stats and error handling

- `fetchStats()`: GET `/api/stats` → `{ version, active_chats, total_memories, ready_documents }`
- `_apiError(res, fallback)`: Extracts `body.error` or `body.detail` from JSON
  error responses; falls back to `"<fallback> (HTTP <status>)"`.

### Upload: why XHR

`uploadDocumentFile` uses `XMLHttpRequest` instead of `fetch()` because
Chainlit's `uploadFile()` flow requires tracking upload progress. The XHR
`upload.onprogress` event provides `loaded/total` byte counts for a progress
bar. `withCredentials: true` ensures auth cookies are sent.

## Vite proxy configuration

`vite.config.js` proxies 11 paths to `http://localhost:8000`:

```js
proxy: {
  "/api":    { target: "http://localhost:8000" },           // REST
  "/ws":     { target: "http://localhost:8000", ws: true }, // WebSocket (Socket.IO)
  "/auth":   { target: "http://localhost:8000" },           // Chainlit auth
  "/login":  { target: "http://localhost:8000" },           // Auto-login POST
  "/logout": { target: "http://localhost:8000" },
  "/user":   { target: "http://localhost:8000" },
  // ... and 5 more Chainlit-specific paths
}
```

The `/ws` proxy is the critical one — it upgrades HTTP to WebSocket for
Chainlit's Socket.IO transport. Without it, the chat would not receive
streamed responses.

## Console-forward plugin

`console-forward-plugin.mjs` is a custom Vite dev-server plugin that forwards
browser `console.log`/`warn`/`error`/`info`/`debug` calls and uncaught
exceptions to the Vite terminal and `logs/browser.log`.

Mechanism:

1. A virtual module (`virtual:console-forward`) injects a browser script that
   intercepts all `console.*` methods.
2. Entries are batched (200ms interval, max 20 per batch) and POSTed to
   `/__console_forward`.
3. The server-side middleware colorizes output and appends to
   `../logs/browser.log`.

This is dev-only (`apply: "serve"`) and has no production impact.

## Active/inactive pattern

Chats, documents, and memories all follow the same soft-delete pattern:

| State | Meaning | Transition buttons |
|---|---|---|
| **active** | Visible, usable, indexed | End / Deactivate → inactive |
| **inactive** | Hidden from normal use but recoverable | Reactivate / Activate → active; Delete → gone |

**Chat lifecycle:**

- **New chat** → `active` (immediately)
- **End chat** → `inactive` (calls `POST /api/chats/:id/end`; triggers backend `chat_end` action including gist generation and memory consolidation)
- **Reactivate** → `active` (calls `POST /api/chats/:id/reactivate`)
- **Fork** → creates a new `active` chat from the current one (server-side copy of messages, fresh session)
- **Delete** → hard delete (calls `DELETE /api/chats/:id`)

**Document lifecycle:**

- **Upload** → `Ready` (active, indexed in Chroma, retrievable)
- **Deactivate** → `deleted` (soft-deleted; not retrieved but Chroma chunks remain)
- **Activate** → `Ready` (restored to active retrieval)
- **Delete** → hard delete (removes metadata; Chroma chunks **not** cleaned up — see `KNOWN_LIMITATIONS.md`)

**Memory lifecycle:**

- **Created** → `active` (LangMem-extracted or API-created)
- **Deactivate** → `deleted` (soft-deleted; not returned by retrieval)
- **Activate** → `active` (restored)
- **Delete** → hard delete (removed from SQLite + Chroma vector index)

The UI consistently renders active items in `bg-surface-container-low` with a
primary-colored accent bar, and inactive items in `bg-surface-container-lowest`
at reduced opacity.

## Trace display

Every assistant message may carry a **workflow trace** embedded as an HTML
comment inside the message output:

```html
<!--breamon-trace:{"retrieved": [...], "saved": [...], "orchestration": "...", "retrieval_errors": [...]}-->
```

The `Message` component in `ChainlitChat.jsx`:

1. **Parses** the comment with a regex: `/<!--breamon-trace:([\s\S]*?)-->/`
2. **Falls back** to `msg.metadata?.trace` if no comment is found (live messages
   may carry trace in metadata rather than the output text)
3. **Strips** the comment from the displayed message text
4. **Renders** the trace sections in a collapsible "system info" panel below
   the message

The collapsible panel shows:

- **Retrieved Memories**: a table of candidates (up to 6 columns)
- **Saved Memories**: a table of newly persisted memories
- **Orchestration**: a text block (orchestration mode, route, budget info)
- **Retrieval Errors**: a bulleted error list

Expand/collapse state is persisted per message ID in `localStorage`
(`breamon-expanded-<msgId>`), surviving page reloads.

Messages rendered specially:

- **User messages**: offset right, almond-silk right border
- **Assistant messages**: purple left border, optional trace panel
- **Error messages**: `error:`-prefix IDs get a red "ERROR:" label and thick
  purple border
- **Indexed messages**: `indexed:`-prefix IDs get a solid purple bar with a
  checkmark icon (used for "X chunks indexed" feedback after upload)

## Authentication flow

Braemon uses Chainlit's cookie-based auth with automatic login for local
prototyping:

```text
main.jsx renders
  → AuthWrapper checks isReady && !isAuthenticated
    → POST /login { username: "local", password: "local" }
      → on success: window.location.reload() to apply the auth cookie
      → on failure: console.error + no retry (user sees spinner forever)
  → Once authenticated, renders <App/>
```

Key details:

- The `ChainlitAPI` client reads auth from the same-origin cookie set by the
  `/login` endpoint.
- The `AuthWrapper` blocks rendering until `isReady && isAuthenticated`, showing
  an "Authenticating..." spinner.
- The auto-login fires exactly once (guarded by the `isReady` check in the
  `useEffect` dependency array `[isReady, isAuthenticated]`).
- A page reload after successful login is necessary so the Chainlit SDK can
  pick up the cookie before the WebSocket connection is established.

In production, this would be replaced with a real login form, but the
architectural pattern (Chainlit auth hook → cookie → SDK reads cookie → WebSocket
auth) remains the same.

## Design system ("Stitch")

Braemon's design system is called **Stitch**. It is dark-only, built on
Tailwind CSS v4 with custom `@theme` tokens.

### Color palette

```text
Background hierarchy (darkest → lightest):
  surface-container-lowest (#0e0e10)  →  surface-dim (#131315)
  → surface (#131315)  →  surface-container-low (#1c1b1d)
  → surface-container (#201f21)  →  surface-container-high (#2a2a2c)
  → surface-container-highest (#353436)  →  surface-bright (#3a393b)

Accent colors (semantic aliases):
  brand-purple  #6b5b95  — primary brand color, borders, accents, spinner
  almond-silk   #c9ada7  — primary interactive elements (New Chat, Send button)
  lilac-ash     #9a8c98  — secondary borders, muted UI elements
  dusty-grape   #4a4e69  — hover backgrounds, subdued containers
  space-indigo  #22223b  — deep container backgrounds

Utility colors:
  primary       #c5c3e4  — text, nav links, section headers
  secondary     #c1c4e5  — secondary text, section labels
  tertiary      #d2c2cf  — tertiary accents
  error         #ffb4ab  — error text and borders
  outline       #928f98  — border and separator lines
```

### Typography

Single font family: **Inter**. Typographic scale via Tailwind custom text tokens:

| Token | Size | Use |
|---|---|---|
| `text-display` | 32px/40px, 600 | Page titles (Home hero, Document Library, Memory Vault) |
| `text-headline-lg` | 24px/32px, 600 | Section headers |

| `text-label-md` | 12px/16px, 600, 0.02em tracking | Section labels, nav items, table headers |
| `text-label-sm` | 11px/14px, 500 | Small labels |
| `text-code` | 13px/20px, 400 | Monospace-like code display (uses Inter, not a mono font) |

### Custom utility classes

- `glass-panel`: Frosted glass effect — `rgba(34,34,59,0.4)` background,
  12px blur, subtle lilac-ash border. Used for Home cards and diagnostic
  panels.
- `bg-gradient-animate`: Animated dark gradient background (15s cycle).
  Applied to the Layout body.
- `custom-scrollbar`: Thin 6px scrollbar with semi-transparent thumb.
- `material-symbols-outlined`: Google Material Symbols icon font configuration.

### Spacing scale

`xs(4px) → sm(8px) → md(12px) → lg(16px) → xl(24px)`, plus `gutter(12px)` and
`margin(20px)` for page-level padding.

## Key design decisions

1. **Chainlit as a library, not a hosted app.** Braemon imports
   `@chainlit/react-client` hooks directly rather than running the Chainlit
   React app. This gives full control over routing, layout, auth, and custom
   pages while keeping the Chainlit chat protocol (WebSocket, streaming,
   file upload).

2. **Auto-login for local dev.** Hardcoded `username: "local"` / `password:
   "local"` eliminates login friction during development. Production would
   need a real auth provider but the cookie-based architecture remains the
   same.

3. **Soft-delete everywhere.** Nothing is truly deleted from the UI by
   default. Chats, documents, and memories move to inactive/archived states
   before hard delete. This gives users a safety net and matches the backend's
   lifecycle semantics.

4. **Trace in HTML comments, not metadata.** Embedding trace JSON as an HTML
   comment inside the message output ensures traces survive message
   serialization through the Chainlit protocol. The backend injects the
   comment before the message is persisted, and the frontend strips it before
   display.

5. **Unguarded Recoil + UUID for session isolation.** Each `ChainlitChat`
   mount gets a fresh UUID session. Together with `key={chatId}` on the
   component, this ensures clean WebSocket teardown and state reset when
   switching between chats.

6. **XHR for file uploads.** The native `fetch()` API does not support upload
   progress events. XHR is used only for document uploads where the progress
   bar is user-visible.
