# src/actions/ — Lifecycle Actions

**TL;DR:** Two explicit user-triggered lifecycle action classes. `ChatEndAction`
(97 lines) finalizes pending memory and gist processing before marking a chat
inactive — the chat transitions only after all processing succeeds; uses
`guarded_chat_operation()` for process-local serialization. `ChatForkAction`
(29 lines) creates an independent transactional copy of a chat — copies
messages, chat gists, and answer inspections; does **not** copy
`chat_memory_state` or per-chat long-term memories; no locking required.

---

## Architecture overview

Lifecycle actions are explicit, user-triggered operations on chats. They are
not background jobs or scheduled tasks — every action happens in response to
a direct API call from the frontend.

```text
API route handler
  │
  ▼
Action.execute(chat_id)
  │  [ChatEndAction only: acquires guarded_chat_operation() lock]
  │  performs deterministic work
  │  commits to database
  ▼
Result (ChatEndResult or new chat_id string)
```

Actions are deterministic service classes, not LLM-backed agents. They use
constructor injection for dependencies (`Database`, memory processor, gist
finalizer, ID factory) so every dependency is replaceable for testing.

## ChatEndAction (`chat_end.py`, 97 lines)

### Purpose

Finalize a chat by processing all pending messages through the memory and
gist pipelines, then mark the chat inactive. The chat is **only** transitioned
to inactive after both pipelines succeed — any failure aborts the transition.

### Pipeline

```text
ChatEndAction.execute(chat_id)
  │
  ├─ 1. Acquire process-local lock (guarded_chat_operation)
  │
  ├─ 2. Memory processing (must succeed)
  │     memory.process_all_for_chat_end(chat_id)
  │     → ChatEndMemoryProcessingResult(processed_message_count, batch_count)
  │     Any exception → logged, re-raised, chat stays active
  │
  ├─ 3. Gist finalization (must succeed)
  │     gist_finalizer.finalize_chat(chat_id)
  │     → PreviousChatGistFinalizationResult(created_count, processed_message_count, batch_count)
  │     Any exception → logged, re-raised, chat stays active
  │
  ├─ 4. Mark chat inactive
  │     database.mark_chat_inactive(chat_id)
  │
  └─ 5. Return ChatEndResult with counts from both pipelines
```

### Dependency protocols

Two `Protocol` classes define required interfaces:

- **`ChatEndMemoryProcessor`**: Requires `process_all_for_chat_end(chat_id) →
  ChatEndMemoryProcessingResult`. In production, implemented by
  `ShortTermMemory`.
- **`ChatEndGistFinalizer`**: Requires `finalize_chat(chat_id) →
  PreviousChatGistFinalizationResult`. In production, defaults to
  `PreviousChatGistGenerator` with a `DeterministicPreviousChatGistExtractor`.

The `gist_finalizer` parameter is optional — if not injected, a
`PreviousChatGistGenerator` is constructed with the deterministic extractor
as fallback. Gist finalization is always attempted at chat end, even when the
caller doesn't explicitly configure it.

### Result

`ChatEndResult` — frozen dataclass:

| Field | Type | Source |
|---|---|---|
| `chat_id` | `str` | The ended chat |
| `processed_message_count` | `int` | From memory result |
| `batch_count` | `int` | From memory result |
| `gist_count` | `int` | Gists created (from `gist_result.created_count`), default `0` |
| `gist_processed_message_count` | `int` | From `gist_result.processed_message_count`, default `0` |
| `gist_batch_count` | `int` | From `gist_result.batch_count`, default `0` |
| `inactive` | `bool` | Always `True` on success |

## ChatForkAction (`chat_fork.py`, 29 lines)

### Purpose

Create an independent transactional copy of an existing chat. The fork is a
brand-new chat with its own ID, containing copies of messages, chat gists,
and answer inspections from the source chat. The forked chat starts
**active** regardless of the source chat's status.

### Implementation

```text
ChatForkAction.execute(chat_id)
  │
  ├─ 1. Generate new chat_id (id_factory() → UUID string)
  │
  ├─ 2. database.fork_chat(source_chat_id, new_chat_id)
  │     Transactionally copies:
  │     • messages (all roles, with summarized=1 for inherited messages)
  │     • chat_gists (with remapped message IDs and chat-local JSON)
  │     • answer_inspections (with remapped message/chat IDs)
  │     NOT copied:
  │     • chat_memory_state (per-chat compatibility cache)
  │     • long_term_memories (per-chat namespace)
  │     User/project-scoped memories remain shared through existing namespaces.
  │     New chat is active by default.
  │
  └─ 3. Return new_chat_id
```

Forking is intentionally simple — no locking, no memory processing, no gist
generation. It's a pure database operation that copies rows with message ID
remapping.

**Forked message semantics:** Inherited (copied) messages are marked
`summarized=1` — they are treated as already semantically processed. Only
post-fork messages can produce new global structured memories.

### ID factory

The `id_factory` parameter defaults to `generate_chat_id()` which returns
`str(uuid4())`. Replaced in tests for deterministic IDs.

## Locking pattern

Only `ChatEndAction` uses `guarded_chat_operation()` from
`src/lifecycle/operation_guard.py`:

```python
with guarded_chat_operation(self.database.path, chat_id):
    return self._execute_locked(chat_id)
```

**How it works:**

- `guarded_chat_operation()` is a context manager that acquires a
  process-local `RLock` keyed by `(database_path, chat_id)`.
- The lock is derived from `chat_operation_lock()` which resolves the
  database path and creates a per-chat `RLock` if one doesn't exist.
- Only **one** operation per chat can execute at a time within the same
  Python process — prevents concurrent `send_message` and `chat_end` on
  the same chat.
- Process-local only — sufficient for single-process Chainlit deployments.

**Which actions use it:**

| Action | Locked? | Why |
|---|---|---|
| `ChatEndAction.execute()` | Yes | Prevents racing with in-flight message processing |
| `ChatForkAction.execute()` | No | Pure read + insert across different chat IDs — no shared mutable state |
| `send_message` (in `chat_service.py`) | Yes | Prevents racing with chat end on the same chat |

## Key design decisions

1. **Chat end is all-or-nothing.** Memory processing and gist finalization
   must both succeed for the chat to transition to inactive. A failure in
   either keeps the chat active and raises an exception to the caller.

2. **Gist finalization is always attempted.** The default constructor
   provides a full gist pipeline, so `ChatEndAction` never silently skips
   gist generation.

3. **Forking is lockless.** Forking reads messages from one chat and inserts
   into another — there is no mutable shared state that requires
   serialization.

4. **Forked messages are pre-processed.** Inherited messages are marked
   `summarized=1`. Post-fork messages are the only ones eligible for new
   structured-memory extraction. Shared user/project memories persist
   across forks unchanged.

5. **Process-local locking only.** The system does not use distributed locks.
   This is intentional for a single-process Chainlit deployment.

6. **Protocols for testability.** `ChatEndMemoryProcessor` and
   `ChatEndGistFinalizer` are `Protocol` classes, not ABCs — structural
   subtyping allows any object with the right method to satisfy the
   interface without explicit inheritance.
