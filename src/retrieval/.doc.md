# Retrieval (`src/retrieval/`)

**TL;DR:** 7 source retrievers → `MemoryCandidate[]` → `MemoryReranker` (11-feature deterministic + optional LLM/cross-encoder) → ranked candidates. Gists are orientation scaffolds; `GistRawSpanExpander` converts them into bounded exact SQLite evidence (`raw_message_span` candidates) before ranking.

## Architecture

The retrieval subsystem implements a two-stage pipeline:

```
RoutePlan.sources
    → RetrieverDispatcher (dispatch to source retrievers)
    → GistRawSpanExpander (gist → raw_span expansion)
    → MemoryReranker (score, rank, trace)
    → ranked MemoryCandidate[]
```

All retrievers follow a single protocol. Every source returns a flat list of `MemoryCandidate` objects. The reranker operates on the unified candidate list; it does not know or care which retriever produced which candidate beyond the `source` field.

## Retriever contract

Every source retriever implements the `SourceRetriever` protocol:

```python
class SourceRetriever(Protocol):
    def retrieve(self, chat_id: str, source_plan: SourcePlan) -> list[MemoryCandidate]:
        ...
```

`SourcePlan` carries the query, limit, filters dict, and enabled flag from the routing decision. Retrievers are free to use or ignore any field — the contract is deliberately narrow.

## `RetrieverDispatcher`

The central hub that owns all retrievers and orchestrates one retrieval pass.

**Construction:** Wires 7 retrievers (named by source string) plus a `GistRawSpanExpander`. Default retrievers are constructed in `__init__`; each can be overridden via the `retrievers` dict for testing.

**`retrieve(chat_id, route_plan)`:**

1. Iterates `route_plan.sources`; skips disabled sources.
2. For `document_memory`, resolves document scope via `DocumentRegistry` before invoking the retriever. If resolution fails with `DocumentScopeError`, records the error and continues.
3. Invokes the matching retriever. If the retriever raises, records the error in `self.last_errors` and continues (fail-soft per source).
4. For `document_memory` queries that returned empty despite having allowed document IDs, retries with a fallback query built from the original query + document names.
5. After all sources are dispatched, runs `GistRawSpanExpander.expand()` on the collected candidates.
6. Returns `candidates + expanded_gist_spans`.

**Environment variables:**

- `DOCUMENT_RETRIEVAL_MODE`: selects the document backend (currently only `langchain_chroma` is supported).

## Retriever catalog

All 7 retrievers implement the same interface. They differ in what they retrieve and which storage backend they query.

| Retriever | Lines | Source | Backend | Default |
|---|---|---|---|---|
| `RecentMessagesRetriever` | 40 | `recent_messages` | SQLite `messages` table, last N messages | Enabled |
| `StructuredMemoryRetriever` | 281 | `structured_memory` | SQLite `long_term_store` + optional Chroma vector index | Enabled |
| `LangChainChromaRetriever` | 280 | `document_memory` | Chroma vector store (`data/chroma/`) | Enabled |
| `RawMessageSpanRetriever` | 850 | `raw_message_span` | SQLite `messages` table (any chat) | Enabled |
| `CurrentChatSpanRetriever` | 266 | `current_chat_span` | SQLite `messages` (current chat only) | Enabled |
| `CurrentChatGistRetriever` | 112 | `current_chat_gist` | SQLite `chat_gists` table (lexical match) | Infrastructure stub |
| `PreviousChatGistRetriever` | 55 | `previous_chat_gist` | SQLite `chat_gists` table (lexical match) | Disabled by default |

### `RecentMessagesRetriever`

Loads the last N messages from the current chat. Each message becomes one `MemoryCandidate` with `source="recent_messages"`, `role`, `created_at`, and `order` metadata. Simple and deterministic.

### `StructuredMemoryRetriever`

Retrieves durable facts, preferences, decisions, corrections, and other structured memories. Supports three modes via `STRUCTURED_MEMORY_RETRIEVAL_MODE`:

- **`sqlite`** (default): Queries `long_term_store` by namespace with optional keyword search. Falls back to `chat_memory_state` (LangMem snapshot) when the store is empty, then supplements with DB-active records not present in the snapshot.
- **`vector`**: Searches the Chroma long-term memory vector index; falls back to sqlite mode on any error.
- **`hybrid`**: Combines sqlite and vector results, deduplicates by namespace + memory_id, returns up to `source_plan.limit` (default 10).

### `LangChainChromaRetriever`

The document chunk retriever. Backed by a LangChain-Chroma vector store at `data/chroma/`. Uses `sentence-transformers/all-MiniLM-L6-v2` embeddings by default.

**Key behaviors:**

- Document filtering: `allowed_document_ids` in `source_plan.filters` scopes retrieval to specific documents.
- Can both index documents (`index_text_document`) and retrieve (`retrieve`).
- Lazy-loads the Chroma vector store and HuggingFace embeddings on first use.
- Raises `LangChainChromaUnavailable` when dependencies are missing.

**Environment variables:**

- `LANGCHAIN_CHROMA_PERSIST_DIR`, `EMBEDDING_MODEL_NAME`, `LANGCHAIN_CHUNK_SIZE`, `LANGCHAIN_CHUNK_OVERLAP`, `DOCUMENT_TOP_K`.

### `RawMessageSpanRetriever`

The most sophisticated retriever (850 lines). Retrieves exact transcript evidence from any chat.

**Three modes:**

1. **Explicit span mode:** triggered when `source_plan.filters` contains `span_chat_id`, `start_message_id`, and `end_message_id`. Loads the exact message range and formats it with anchor-aware truncation.

2. **Direct mode:** cross-chat lexical search. Scores messages across inactive chats by query term overlap, selects the best scoring spans, expands each span by a configurable window, and formats them. Supports `context_profile=global_summary` for chronological chunked retrieval.

3. **Global summary mode:** activated by `context_profile=global_summary`. Returns raw messages from inactive chats in reverse-chronological order, chunked into bounded segments.

**Anchor preservation:** When truncating spans to `max_chars` (default 4000), the retriever preserves messages containing query terms as "anchors" and marks truncation boundaries with explicit markers (`[raw message span truncated before anchor]`, etc.).

### `CurrentChatSpanRetriever`

Same-chat lexical span extraction. Scores messages in the current chat by query term overlap, identifies the top-scoring spans, expands each span by a configurable context window (`window_messages`, default 2), merges overlapping spans, and formats the result as raw message evidence.

Excludes the most recent messages (which `RecentMessagesRetriever` already covers) and the query message itself to avoid redundancy.

### `CurrentChatGistRetriever`

Infrastructure stub for current-chat gist retrieval. Reads stored gists from SQLite `chat_gists` when they exist, performs temporary lexical filtering by query terms. Does **not** generate gists or run vector search — those responsibilities belong to `ChatGistSummarizer` in the memory subsystem.

### `PreviousChatGistRetriever`

Disabled-by-default retriever for previous-chat gists. Reads gists of type `previous_chat_gist` from SQLite, filters by query overlap, and returns up to `source_plan.limit` (default 4). Supports `context_profile=global_summary` for chronological ordering.

**Why disabled by default:** Previous-chat gist generation is off by default. When gists don't exist, this retriever returns nothing.

## Gist → raw span expansion

Gists are **orientation** — they summarise what a chat segment was about. But answers need **proof** — exact transcript evidence. `GistRawSpanExpander` bridges this gap.

**Pattern:**

```
Gist candidates (current_chat_gist or previous_chat_gist)
    → Resolve start/end message IDs from gist metadata or persisted gist record
    → Merge overlapping gist ranges (same chat_id, contiguous)
    → Load raw messages from SQLite for each range
    → Select a bounded window around the query-best message (max 12 messages)
    → Identify anchor messages (best query-term matches)
    → Format with anchor-aware character truncation (max 4000 chars)
    → Emit as raw_message_span MemoryCandidates
```

**Key design decisions:**

- Expansion is **greedy and bounded**: max 12 messages per span, max 4000 chars per formatted output.
- Overlapping gist ranges are **merged** before expansion — avoids duplicate evidence.
- Each expanded candidate carries full provenance: `parent_gist_id`, `derived_from_source`, `original_start/end_message_id`, `selection_reason`, `truncated` flag.
- Expansion happens **after all retrievers run but before reranking** — so expanded raw spans participate in the same scoring pass.

## MemoryReranker

The query-aware ranking engine. Operates in four modes controlled by `RERANKER_MODE`:

| Mode | Behavior | Requires |
|---|---|---|
| `deterministic` (default) | 11 hand-crafted features, weighted sum, stable sort | Nothing |
| `cross_encoder` | Deterministic top-k, then sentence-transformers cross-encoder rescore + combine | `sentence-transformers` |
| `hybrid` | Cross-encoder rescore, then gated LLM rerank if ambiguous | `sentence-transformers` + LLM model |
| `llm` | Deterministic scoring, then listwise LLM reranking | LLM model |

### Deterministic scoring (11 features)

Every candidate receives a score from a weighted sum of 11 features:

| Feature | Weight | What it measures |
|---|---|---|
| `lexical_overlap` | 0.35 | Query term coverage in candidate content (stopword-filtered) |
| `query_source_boost` | 0.25 | Source-specific boost when query explicitly references that source type |
| `semantic_score` | 0.20 | Vector/semantic relevance from candidate metadata |
| `similarity_score` | 0.20 | Similarity from candidate metadata (defaults to 0.5) |
| `importance` | 0.15 | Importance from candidate metadata (defaults to 0.5) |
| `confidence` | 0.15 | Confidence from metadata, score, or 0.5 default |
| `recency` | 0.10 | 1/(order+1) when order available, else metadata |
| `usage_count` | 0.05 | Normalized usage count / 10 |
| `source_priority` | 0.10 | Static source priority from policy |
| `status_penalty` | 1.0 | Negative penalty: archived=0.15, superseded=0.7, deleted=1.0 |
| `redundancy_penalty` | 1.0 | Exact duplicate content penalty = 0.08 per duplicate |

Candidates are sorted by `( -score, original_rank )` for stable tie-breaking. Every candidate receives a `score_breakdown` in metadata with feature values, weights, contributions, and final score.

### Source priority tiers

Static source priorities (configurable via `RerankerPolicy`):

```
structured_memory: 0.95    (highest — user facts/preferences)
recent_messages:   0.90
current_chat_gist: 0.72
previous_chat_gist: 0.65
document_memory:   0.60
raw_message_span:  0.58    (lowest — raw evidence, only useful when relevant)
```

### Cross-encoder mode

1. Take deterministic top-k (default 10).
2. Score each against the query using `sentence-transformers` CrossEncoder (`BAAI/bge-reranker-v2-m3` by default).
3. Combine: `combined = CE_weight × CE_score + (1-CE_weight) × normalized_deterministic_score` (CE_weight default 0.65).
4. Re-sort the top-k; append untouched tail candidates.

### Hybrid mode

Cross-encoder rescore → **LLM gate decision** → optional listwise LLM rerank.

The gate opens (triggers LLM reranking) when:

- **Provenance conflict:** top candidates include both `previous_chat_gist` and `raw_message_span` AND the query asks for exact evidence.
- **Old decision conflict:** top candidates include both `previous_chat_gist` and `raw_message_span` AND the query asks about earlier decisions.
- **Cross-source ambiguity** with small top-margin or ranking disagreement after cross-encoder.

The gate stays closed (reuses deterministic/cross-encoder order) when:

- Only one candidate.
- All top candidates share the same source.
- Top score margin exceeds `ambiguity_margin` (default 0.15) and rankings agree.

### LLM reranking

The LLM receives a system prompt instructing it to rank by usefulness, prefer grounded provenance for evidence queries, and avoid distractors. It returns JSON with `ranked_candidate_ids`, `confidence`, and `reason`.

**Safety:** The LLM result is validated — candidate IDs must match the rerank pool, no duplicates, no unknown IDs, numeric confidence. If confidence < `llm_min_confidence` (0.55), the LLM order is discarded and deterministic ordering is used. On any exception, falls back to deterministic.

### Trace metadata

Every `rank_with_trace` call produces rich metadata:

- `reranker_mode`, `fallback_used`, `fallback_reason`
- Per-candidate `score_breakdown` (features, weights, contributions)
- `deterministic_top_margin`, `post_cross_encoder_top_margin`
- `top_candidate_sources` (unique sources in top 3)
- Cross-encoder scores, combined scores (when used)
- LLM decision gate, ranked IDs, confidence, reason (when used)
- `final_ranks` for every candidate

## Cross-encoder backend

`SentenceTransformersCrossEncoderBackend` is a lazy-loading adapter. It loads the model only on first `score()` call, avoiding startup delay when `RERANKER_MODE=deterministic`. Logit scores are sigmoid-normalized to [0,1] when they fall outside that range.

## Error handling

- **Per-source fail-soft:** `RetrieverDispatcher` catches exceptions from individual retrievers, records them in `self.last_errors`, and continues to the next source.
- **Document scope errors:** `DocumentScopeError` from `DocumentRegistry` is caught before retriever invocation; no retriever is called when scope resolution fails.
- **Gist expansion is best-effort:** If `database.messages_for_chat_span` returns empty or no messages survive bounding, that gist is silently skipped.
- **Reranker fallbacks:** Cross-encoder errors revert to deterministic ordering. LLM errors revert to previous stage ordering. All fallbacks are recorded in trace metadata.
