# src/routing/ â€” Routing

**TL;DR:** Two deterministic query-routing systems. The production path (`QueryAnalyzer` â†’ `RoutePlanner` â†’ `RoutingAgent`) classifies intents from 7 boolean lexical signals and enables memory sources accordingly. An alternate `SemanticRouter` (default-off) provides richer intent classification with evidence contracts and temporal scopes. Both produce `RoutePlan` objects consumed by the same `RetrieverDispatcher`. Fallback is always conservative: recent messages + structured memory enabled, document memory off.

---

## Why two routing systems?

The codebase has two routers because they serve different ambitions:

| Router | Path | Status | Strengths |
|---|---|---|---|
| **Rule-based** (production) | `QueryAnalyzer` â†’ `RoutePlanner` â†’ `RoutingAgent` | Active, default | Deterministic, simple lexical signals, 4 lines of confidence logic, predictable fallback |
| **Semantic** (evaluation) | `SemanticRouter` | Default-off, available via `SEMANTIC_ROUTER_ENABLED` | Richer intent taxonomy (8 types), temporal scopes, `EvidenceContract`, per-source `RetrievalQuery` hints |

The production router is intentionally simple â€” it uses 7 boolean flags detected via substring matching (`"this chat"`, `"previous chat"`, `"document"`, `"decide"`, etc.). This keeps routing fast, deterministic, and debuggable. The semantic router was built as a research/evaluation baseline that models finer-grained query understanding without LLM involvement. Both produce the same output contract (`RoutePlan`) and plug into the same downstream pipeline, so switching is a configuration change, not a code change.

The semantic router is default-off for two reasons:

1. Its evidence contracts (`requires_raw_span`, `must_not_answer_from_gist_only`) are not yet enforced by the context manager â€” the contract is emitted but not consumed downstream.
2. Its richer source selection (separating `current_chat_span` from `previous_chat_gist` from `raw_message_span`) requires the retriever infrastructure to have full per-source implementations, some of which are still stubs.

### QueryAnalyzer: deterministic signal detection

`QueryAnalyzer` is a `rule-light` analyzer that produces 7 boolean `QuerySignals` via substring matching against hardcoded term lists in `QueryAnalyzerPolicy`:

| Signal | Triggers on substring like | Enables source |
|---|---|---|
| `asks_about_current_chat` | "this chat", "we discussed", "earlier" | `current_chat_span` (when no previous-chat signal) |
| `asks_about_previous_memory` | "previous chat", "last time" | `previous_chat_gist`, `raw_message_span` |
| `asks_about_documents` | "document", "pdf", "upload", "according to" | `document_memory` |
| `asks_about_decision` | "decide", "choose", "which model" | Triggers `memory_recall` profile only |
| `asks_about_task` | "task", "todo", "next step" | Triggers `memory_recall` profile only |
| `asks_for_exact_quote` | "exact words", "verbatim", "quote me" | Metadata-only (not source-gating yet) |
| `asks_for_global_summary` | "summarize the book", "summarize previous" | Forces `context_profile=global_summary` |

The `detect_intent()` function maps signals to a single coarse intent (first-match priority: global_summary â†’ document â†’ previous_memory â†’ decision â†’ task â†’ current_chat â†’ general).

Intent-to-context-profile mapping lives in `RoutePlannerPolicy.intent_context_profiles`:

- `general_question` â†’ `general_chat`
- All memory-oriented intents â†’ `memory_recall`
- `document_question` â†’ `document_question`

### RoutePlanner: from analysis to source plans

`RoutePlanner` owns the central `RoutePlannerPolicy` that declares:

- **`active_sources`**: `recent_messages` (always on), `structured_memory` (always on)
- **`future_sources`**: 6 conditional sources enabled per-signal (document, spans, gists, legacy aliases)
- **Default policies**: `ranking_profile="none_current_order"`, `fallback_policy`, `update_policy`, `termination_policy` â€” carried through to `RoutePlan`

The `simplify_retrieval_query()` call (controllable via `ENABLE_RETRIEVAL_QUERY_SIMPLIFICATION`) applies conservative rewrites:

1. Preserves negations, quoted phrases, and multiple-choice options (no rewrite).
2. For `global_summary` profile: rewrites to "global summary complete {label} chronological content".
3. Strips leading boilerplate ("based on all the information above...").
4. Entity comparison detection (age/relation questions).

### RoutingAgent: wrapping RoutePlanner

`RoutingAgent` adds two layers on top of `RoutePlanner`:

**Fallback behavior.** Any error in routing â€” missing sources, exception during planning, parse failure â€” produces a conservative `RoutingDecision` with `recent_messages=on`, `structured_memory=on`, `document_memory=off`, `confidence=0.0`, `fallback_mode=True`.

**LLM/Hybrid modes.** When `ROUTING_MODE` is `llm` or `hybrid`, the agent calls a chat model with a structured-output prompt. In `hybrid` mode, the deterministic baseline always enables `recent_messages` and `structured_memory`; the LLM only decides whether `document_memory` is useful. Any LLM result with `confidence < 0.5` triggers fallback to rule routing.

`RoutingDecision.to_trace_dict()` produces the compact trace payload consumed by `WorkflowTrace` metadata â€” including active/disabled sources and routing mode info.

## SemanticRouter

`SemanticRouter` is a richer deterministic router that classifies queries into 8 intents using ~60 regex patterns:

| Intent | Confidence | Key pattern examples | Temporal scope |
|---|---|---|---|
| `EXACT_QUOTE` | 0.95 | "quote me", "exact phrase", "verbatim" | `ANY_CHAT` |
| `DOCUMENT_QA` | 0.92 | "according to the document", "the uploaded file" | `DOCUMENTS` |
| `STRUCTURED_PREFERENCE_RECALL` | 0.90 | "my preference", "do i prefer" | `GLOBAL_STRUCTURED_MEMORY` |
| `PREVIOUS_CHAT_RECALL` | 0.88 | "what did we discuss before", "last time" | `PREVIOUS_CHATS` |
| `SAME_CHAT_RECALL` | 0.88 | "what did i say earlier", "earlier in this chat" | `CURRENT_CHAT` |
| `PROJECT_STATE_SUMMARY` | 0.86 | "where are we", "project state", "open tasks?" | `ANY_CHAT` â†’ scopes: durable + previous |
| `MEMORY_QA` | 0.82 | Factual question with first-person cues, requires retrieval | `ANY_CHAT` |
| `CASUAL_CHAT` | 0.70 | No memory/document pattern matched | `NONE` |

### Key differences from production routing

1. **Evidence contracts.** `EvidenceContract` declares what evidence the answer requires: `requires_raw_span`, `requires_document_citation`, `requires_structured_memory`, `must_not_answer_from_gist_only`. These are emitted on the `SemanticRoutePlan` but not yet enforced downstream.

2. **Temporal scopes.** `CURRENT_CHAT`, `PREVIOUS_CHATS`, `ANY_CHAT`, `DOCUMENTS`, `GLOBAL_STRUCTURED_MEMORY`, `NONE` â€” independent of intent. Used to compute `required_scopes` (a `frozenset[MemoryScope]`) which selects memory sources via `sources_for()`.

3. **Per-source retrieval queries.** The semantic router can generate multiple `RetrievalQuery` hints with different `allowed_sources` per query. For `EXACT_QUOTE`, it adds a `quote_orientation_search` query scoped to `previous_chat_gist` only.

4. **Task context overrides.** When `task_context="document_qa"`, the intent is forced to `DOCUMENT_QA` with confidence 0.95. This enables same-turn attachment routing.

5. **Adaptation layer.** `SemanticRouter.to_route_plan()` converts the richer `SemanticRoutePlan` into a standard `RoutePlan` consumable by the existing `RetrieverDispatcher`. The adaptation preserves semantic metadata in `RoutePlan.metadata` (router version, language, temporal scope, retrieval need, memory scope).

## Retrieval query simplification

`retrieval_query.py` provides a deterministic function `simplify_retrieval_query()` used by both routers. It never changes the user-visible query â€” only the retrieval query sent to memory sources.

**Preservation rules** (no rewrite):

- Negations detected: "not", "never", "without" â†’ preserves full query
- Quoted phrases: `"exact text"` â†’ preserves
- Multiple-choice patterns: `A) ... B) ...` â†’ preserves

**Rewrite rules** (applied when safe):

- `global_summary` profile â†’ rewrites to "global summary complete {label} chronological content"
- Leading boilerplate removal: strips "based on all the information above..."
- Entity comparison: "Who is older, Alice or Bob?" â†’ "Alice age born Bob age born older"

Controlled by `ENABLE_RETRIEVAL_QUERY_SIMPLIFICATION` (default: enabled). Disable to always use the raw query for retrieval.

## Configuration

| Env variable | Default | Effect |
|---|---|---|
| `ROUTING_MODE` | `"rule"` | `"rule"`, `"llm"`, or `"hybrid"`. Invalid values fall back to `"rule"`. |
| `ENABLE_RETRIEVAL_QUERY_SIMPLIFICATION` | `"1"` | Set to `"0"`/`"false"` to disable retrieval query simplification. |
| `PREVIOUS_CHAT_GIST_RETRIEVAL_ENABLED` | `"1"` | Gates `previous_chat_gist` and `raw_message_span` source enablement in `RoutePlanner`. |
| `SEMANTIC_ROUTER_ENABLED` | unset | When truthy, the `CoordinatorAgent` uses `SemanticRouter` instead of `RoutingAgent`. |
| `GIST_RETRIEVAL_CANDIDATES` | `8` | Candidate limit for `previous_chat_gist` in semantic router. |
| `DIRECT_RAW_RETRIEVAL_CANDIDATES` | `12` | Candidate limit for `raw_message_span` in semantic router. |
