# src/routing/ — Routing

**TL;DR:** Two deterministic query-routing systems. The production path (`QueryAnalyzer` → `RoutePlanner` → `RoutingAgent`) classifies intents from 8 boolean lexical signals and enables memory sources accordingly. An alternate `SemanticRouter` (default-off, used in LangGraph orchestration) provides richer intent classification with evidence contracts and temporal scopes. Both produce `RoutePlan` objects consumed by the same `RetrieverDispatcher`. Fallback is always conservative: recent messages + structured memory enabled, document memory off.

---

## Why two routing systems?

The codebase has two routers because they serve different ambitions:

| Router | Path | Status | Strengths |
|---|---|---|---|
| **Rule-based** (production) | `QueryAnalyzer` → `RoutePlanner` → `RoutingAgent` | Active, default | Deterministic, simple lexical signals, 4 lines of confidence logic, predictable fallback |
| **Semantic** (evaluation) | `SemanticRouter` | Default-off, used only within LangGraph orchestration | Richer intent taxonomy (8 types), temporal scopes, `EvidenceContract`, per-source `RetrievalQuery` hints |

The production router is intentionally simple — it uses 8 boolean flags detected via substring matching against hardcoded term lists in `QueryAnalyzerPolicy`. This keeps routing fast, deterministic, and debuggable. The semantic router was built as a research/evaluation baseline that models finer-grained query understanding without LLM involvement. Both produce the same output contract (`RoutePlan`) and plug into the same downstream pipeline.

The semantic router is default-off for two reasons:

1. Its evidence contracts (`requires_raw_span`, `must_not_answer_from_gist_only`) are not yet enforced by the context manager.
2. Its richer source selection (separating `current_chat_span` from `previous_chat_gist` from `raw_message_span`) requires full per-source implementations, some still stubs.

## Production routing flow

```
User query (string)
  │
  ▼
QueryAnalyzer.analyze(query)
  │  normalize_query() → lowercase, collapse whitespace
  │  contains_any() → substring match against 7 term sets
  │  detect_intent(signals) → "general_question" | "document_question" |
  │      "current_chat_question" | "previous_memory_question" |
  │      "decision_question" | "task_question"
  │  confidence_for(signals) → 0.55 (0 signals) | 0.7 (1 signal) | 0.8 (2+ signals)
  │
  ▼ QueryAnalysis (normalized_query, intent, signals, confidence)
  │
  ▼
RoutePlanner.plan_from_analysis(analysis)
  │  resolve context_profile from intent (or "global_summary" override)
  │  simplify_retrieval_query() → optionally strip boilerplate,
  │      preserve negations/quotes/options, rewrite global summaries
  │  loop over SourceRoutingPolicy tuples → SourcePlan per source
  │  source_enabled() gating:
  │    - document_memory → only when asks_about_documents
  │    - current_chat_span → only current-chat recall (no previous signal)
  │    - previous_chat_gist → PREVIOUS_CHAT_GIST_RETRIEVAL_ENABLED + prev signals
  │    - raw_message_span → same gating as previous_chat_gist
  │    - others → static policy defaults (recent_messages=on, structured_memory=on)
  │
  ▼ RoutePlan (query, intent, confidence, sources[], context_profile, metadata)
  │
  ▼
RoutingAgent.route(query)
  │  mode == "rule" → _rule_decision() → wraps RoutePlanner.plan()
  │  mode == "llm" → _llm_or_fallback_decision()
  │  mode == "hybrid" → LLM decides document_memory only; recent/structured
  │                     forced on from rule baseline
  │
  ▼ RoutingDecision (route_plan + convenience booleans + to_trace_dict())
```

### QueryAnalyzer: deterministic signal detection

`QueryAnalyzer` is a `rule-light` analyzer that produces 8 boolean `QuerySignals`. Seven are detected via substring matching against hardcoded term lists in `QueryAnalyzerPolicy`; the 8th (`asks_general_question`) is auto-computed as `True` when all 7 others are `False`.

| Signal | Triggers on substring like | Enables source |
|---|---|---|
| `asks_about_current_chat` | "this chat", "we discussed", "earlier", "above" | `current_chat_span` (when no previous-chat signal) |
| `asks_about_previous_memory` | "previous chat", "last time", "remember from before" | `previous_chat_gist`, `raw_message_span` |
| `asks_about_documents` | "document", "pdf", "file", "upload", "paper", "source", "according to", and 25 more terms | `document_memory` |
| `asks_about_decision` | "decide", "choose", "which database", "which model" | Triggers `memory_recall` profile only |
| `asks_about_task` | "task", "todo", "next step", "open item" | Triggers `memory_recall` profile only |
| `asks_for_exact_quote` | "exact phrase", "exact words", "verbatim", "quote me" | Metadata-only (not source-gating yet) |
| `asks_for_global_summary` | "summarize the book", "summarize our previous conversation" | Forces `context_profile=global_summary` |
| `asks_general_question` | Auto-computed: True when all 7 above are False | No source toggle (default routing) |

`detect_intent()` maps signals to a single coarse intent with this priority order:

1. `asks_for_global_summary` → `"previous_memory_question"`
2. `asks_about_documents` → `"document_question"`
3. `asks_about_previous_memory` → `"previous_memory_question"`
4. `asks_about_decision` → `"decision_question"`
5. `asks_about_task` → `"task_question"`
6. `asks_about_current_chat` → `"current_chat_question"`
7. Fallthrough → `"general_question"`

Intent-to-context-profile mapping in `RoutePlannerPolicy.intent_context_profiles`:

- `general_question` → `general_chat`
- All memory-oriented intents → `memory_recall`
- `document_question` → `document_question`

Additionally, `inline_summary` detection (queries matching `^summarize this text:`) suppresses the document signal even when document terms are present. This prevents "summarize this text: ..." from triggering document retrieval.

### RoutePlanner: from analysis to source plans

`RoutePlanner` owns the central `RoutePlannerPolicy`:

- **`active_sources`**: `recent_messages` (always on), `structured_memory` (always on)
- **`future_sources`**: 7 conditional sources (current_chat_gist, current_chat_span, previous_chat_gist, raw_message_span, current_chat_chunks, previous_chat_memory, document_memory)
- **Default policies**: `ranking_profile="none_current_order"`, `fallback_policy`, `update_policy`, `termination_policy`

`simplify_retrieval_query()` is called per-plan (controlled by `ENABLE_RETRIEVAL_QUERY_SIMPLIFICATION`). The rewrite result is stored in both `SourcePlan.filters` and `RoutePlan.metadata`.

### RoutingAgent: wrapping RoutePlanner (3 modes)

`RoutingAgent` constructor default is `"rule"`, but the production default from `settings.py` is `"hybrid"`.

**Rule mode** (`ROUTING_MODE=rule`): delegates to `_rule_decision()` → `RoutePlanner.plan()`. On any failure (exception, empty sources), falls back to a conservative fallback decision.

**LLM mode** (`ROUTING_MODE=llm`): calls `_llm_or_fallback_decision()` → structured-output chat model. The LLM selects from `LLM_ROUTABLE_SOURCES` (7 sources). If `confidence < MIN_LLM_ROUTING_CONFIDENCE` (0.5), or any JSON parse error, or missing model — falls back to rule routing with `routing_fallback_reason` set.

**Hybrid mode** (`ROUTING_MODE=hybrid`): deterministic baseline + LLM expansion. The LLM can only **add** sources (`base_enabled_sources | selected_sources`), never remove them. This prevents the LLM from disabling structured memory or recent messages.

**Fallback behavior** (all modes): errors produce `RoutingDecision` with `recent_messages=on`, `structured_memory=on`, `document_memory=off`, `confidence=0.0`, `fallback_mode=True`.

`RoutingDecision.to_trace_dict()` produces the compact trace payload for `WorkflowTrace` — active/disabled sources, intent, context_profile, routing mode, fallback info.

**LLM routing protocol:** `RoutingModel` is a `Protocol` requiring only `chat(messages, temperature) -> str`. The LLM prompt is built by `llm_routing_messages()`. Responses are parsed by `parse_routing_json()` (handles fenced blocks). Legacy three-boolean format (`use_recent_messages`, `use_structured_memory`, `use_document_memory`) is supported via `legacy_selected_sources_from_llm_payload()`.

## SemanticRouter

`SemanticRouter` is a richer deterministic router classifying queries into 8 intents using ~60 regex patterns:

| Intent | Confidence | Key pattern examples | Temporal scope |
|---|---|---|---|
| `EXACT_QUOTE` | 0.95 | "quote me", "exact phrase", "verbatim" | `ANY_CHAT` |
| `DOCUMENT_QA` | 0.92 | "according to the document", "the uploaded file" | `DOCUMENTS` |
| `STRUCTURED_PREFERENCE_RECALL` | 0.90 | "my preference", "do i prefer" | `GLOBAL_STRUCTURED_MEMORY` |
| `PREVIOUS_CHAT_RECALL` | 0.88 | "what did we discuss before", "last time" | `PREVIOUS_CHATS` |
| `SAME_CHAT_RECALL` | 0.88 | "what did i say earlier", "earlier in this chat" | `CURRENT_CHAT` |
| `PROJECT_STATE_SUMMARY` | 0.86 | "where are we", "project state", "open tasks?" | `ANY_CHAT` → scopes: durable + previous |
| `MEMORY_QA` | 0.82 | `CASUAL_CHAT` with factual cues + question pattern → auto-upgraded | `ANY_CHAT` |
| `CASUAL_CHAT` | 0.70 | No memory/document pattern matched | `NONE` |

`MEMORY_QA` is special: it's never directly classified by `classify_intent()`. Instead, when `classify_intent()` returns `CASUAL_CHAT` but `retrieval_need_for()` returns `RETRIEVAL_REQUIRED` (first-person cues + question pattern), the intent is upgraded to `MEMORY_QA` with confidence 0.82.

### Key differences from production routing

1. **Evidence contracts.** `EvidenceContract` declares: `requires_raw_span`, `requires_document_citation`, `requires_structured_memory`, `must_not_answer_from_gist_only`. Emitted on `SemanticRoutePlan`, not yet enforced downstream.

2. **Temporal scopes.** `CURRENT_CHAT`, `PREVIOUS_CHATS`, `ANY_CHAT`, `DOCUMENTS`, `GLOBAL_STRUCTURED_MEMORY`, `NONE` — independent of intent. Computes `required_scopes` (a `frozenset[MemoryScope]`) used by `sources_for()`.

3. **Per-source retrieval queries.** Multiple `RetrievalQuery` hints with different `allowed_sources`. For `EXACT_QUOTE`, adds a `quote_orientation_search` query scoped to `previous_chat_gist`.

4. **Task context overrides.** `task_context="document_qa"` → forced `DOCUMENT_QA`, confidence ≥0.95. Enables same-turn attachment routing.

5. **Adaptation layer.** `SemanticRouter.to_route_plan()` converts `SemanticRoutePlan` → standard `RoutePlan`. Preserves semantic metadata in `RoutePlan.metadata` (router_version, language, temporal_scope, retrieval_need, memory_scope, primary_scope, required_scopes).

### `retrieval_need_for()` — Retrieval necessity logic

Three-tier classification independent of intent:

- `RETRIEVAL_REQUIRED`: non-CASUAL_CHAT intent, summary requests with referential targets, task_context overrides, or question patterns with first-person/memory cues.
- `RETRIEVAL_POSSIBLE`: question patterns without memory cues.
- `RETRIEVAL_NONE`: non-retrieval patterns (greetings, creative writing), or no question/memory cues.

## Retrieval query simplification

`retrieval_query.py` provides two functions:

### `simplify_retrieval_query()` — core rewrite engine

Never changes the user-visible query — only the retrieval query sent to memory sources.

**Preservation rules** (no rewrite):

- Negations detected: "not", "never", "without", "except", "excluding", "neither" → preserves full query
- Quoted phrases: `"exact text"` → preserves
- Multiple-choice patterns: `A) ... B) ...` → preserves
- Disabled or empty: returns no-op

**Rewrite rules** (applied when safe):

- `global_summary` profile → `"global summary complete {label} chronological content"` (detects target: book, conversation, story, novel, document, report)
- Leading boilerplate removal strips "based on all the information above...", "using all the context above...", "according to all the information above..."
- Entity comparison: `"Who is older, Alice or Bob?"` → `"Alice age born Bob age born older"`

Controlled by `ENABLE_RETRIEVAL_QUERY_SIMPLIFICATION` (default: enabled). Disable to always use the raw query.

### `retrieval_query_for_reranking()` — reranker query extraction

Extracts the deterministic retrieval query from `RoutePlan.metadata["retrieval_query"]` for use during reranking. Returns the `fallback` parameter if the metadata key is missing or empty. Used by the reranker to apply query-aware scoring without rewriting the user's visible query.

## Configuration

| Env variable | Default | Effect |
|---|---|---|
| `ROUTING_MODE` | `"hybrid"` | `"rule"`, `"llm"`, or `"hybrid"`. Invalid values fall back to `"rule"`. Production default from `settings.py`. |
| `ORCHESTRATION_MODE` | `"native"` | `"langgraph_demo"` enables SemanticRouter path. `"native"` skips it. |
| `ENABLE_RETRIEVAL_QUERY_SIMPLIFICATION` | `"1"` | Set to `"0"`/`"false"` to disable retrieval query simplification. |
| `PREVIOUS_CHAT_GIST_RETRIEVAL_ENABLED` | `"1"` | Gates `previous_chat_gist` and `raw_message_span` source enablement in `RoutePlanner`. Default `True` when unset. |
| `GIST_RETRIEVAL_CANDIDATES` | `8` | Candidate limit for `previous_chat_gist` in semantic router. |
| `DIRECT_RAW_RETRIEVAL_CANDIDATES` | `12` | Candidate limit for `raw_message_span` in semantic router. |
