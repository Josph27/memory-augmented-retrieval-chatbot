# src/lifecycle/ ŌĆö Operation Guard

**TL;DR:** Process-local locking and idempotent operation keys. `guarded_chat_operation()` serializes state transitions (send, upload, end) per chat in a single application process. The `operation_results` table provides database-level idempotency for uploads and other write-once operations via `record_operation_once()` and `claim_document_upload()`.

---

## Design

Two complementary mechanisms prevent duplicate or overlapping state transitions:

1. **Process-local locks** (`operation_guard.py`): Thread-safe `RLock` per chat, acquired via `guarded_chat_operation()`. Serializes concurrent writes within one application process.

2. **Database idempotency keys** (`database.py` `operation_results` table): `INSERT OR IGNORE` on a unique `operation_id` guarantees at-most-once semantics across restarts.

Together they provide best-effort write safety without a distributed lock manager.

---

## Operation guard (`operation_guard.py`)

### `chat_operation_lock(database_path, chat_id) ŌåÆ RLock`

Returns a process-local `RLock` keyed by `(database_path, chat_id)`. Locks are lazily created and cached in `_CHAT_LOCKS`. Thread-safe via `_GUARD_LOCK` (a module-level `RLock` protecting the lock map).

### `guarded_chat_operation(database_path, chat_id) ŌåÆ Iterator[None]`

Context manager that acquires the per-chat lock for the duration of the block:

```python
with guarded_chat_operation(db_path, chat_id):
    # send message, upload file, end chat, etc.
```

Multiple operations on the same chat within one process are serialized. Operations on different chats proceed concurrently.

**Scope:** Process-local only. If multiple `app.py` processes run simultaneously, locks do not coordinate across them. The database idempotency keys (below) serve as the cross-process safety net.

## Database idempotency (`operation_results` table)

Schema from `database.py:225`:

| Column | Type | Purpose |
|---|---|---|
| `operation_id` | `TEXT PK` | Unique operation key ŌĆö PK enforces at-most-once |
| `operation_type` | `TEXT` | Operation category (e.g. `"document_upload"`) |
| `scope_id` | `TEXT` | Scoping key ŌĆö usually `chat_id` |
| `result_ref` | `TEXT` | Stable result reference ŌĆö usually `document_id` |
| `created_at` | `TEXT` | Timestamp |

### `record_operation_once(operation_id, operation_type, scope_id?, result_ref?) ŌåÆ bool`

Low-level idempotency gate. Returns `True` on first insert, `False` if the `operation_id` already exists. Uses `INSERT OR IGNORE` ŌĆö atomic and safe under concurrent writers.

### `claim_document_upload(operation_id, chat_id, document_id, file_name, source?) ŌåÆ (bool, str)`

Atomically claims a document upload key and creates the initial `document_records` row with status `Uploading`. Two-phase within a single transaction:

1. `INSERT OR IGNORE` into `operation_results` for the upload key.
2. If first insert: create `document_records` row. Return `(True, document_id)`.
3. If duplicate: verify same `chat_id` and `operation_type`. Return `(False, existing_document_id)` for safe retry.
4. If `operation_id` belongs to a different scope/chat: raise `RuntimeError`.

This enables safe client retry ŌĆö the client can re-send the same upload with the same `operation_id` and get back the previously created `document_id`.

### `get_operation_result(operation_id) ŌåÆ StoredOperationResult | None`

Lookup for previously stored operation metadata. Returns `None` for unknown keys.

## Usage in the codebase

### Upload idempotency (`api_routes.py`)

Document upload uses `claim_document_upload()` with an `operation_id` derived from the upload request (e.g., file hash + chat ID). If the client retries a failed upload, the same `operation_id` returns the previously created `document_id` without re-creating metadata.

### Chat operations (`chat_service.py`, `chainlit_data_layer.py`)

`guarded_chat_operation()` wraps critical chat state transitions ŌĆö message send, file upload association, and chat end ŌĆö ensuring they don't interleave within the same process.

---

## Limitations

- **Process-local only.** Locks do not span processes. Multi-process deployments rely on SQLite's built-in WAL serialization + `operation_results` idempotency.
- **No timeout.** `RLock` blocks indefinitely ŌĆö there is no deadlock detection or timeout mechanism.
- **No cleanup.** Lock objects accumulate in `_CHAT_LOCKS` for the lifetime of the process. Memory impact is negligible for typical chat counts.
