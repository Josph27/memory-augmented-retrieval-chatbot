# src/context/ — Deterministic Context Pipeline

**TL;DR:** Five deterministic layers transform ranked `MemoryCandidate` objects
into model-ready messages under a shared token budget. No LLM negotiation.
Layer 0 resolves the effective context window. Layer 1 sets a route-specific
working-memory cap. Layer 2 splits that cap across sources via allocation
profiles. Layer 3 selects candidates with requirement-first evidence
enforcement, deduplication, and marginal-utility ordering. Layer 4 assembles a
`ContextPacket` with validated model-shaped messages.

```
Layer 0: ModelProfile         → ResolvedContextWindow
           resolve_context_window() — min(native, endpoint, app cap)
               │
Layer 1: DynamicWorkingMemoryBudgetPlanner  → DynamicBudgetPlan
           plan() — route-aware cap with required-evidence expansion
               │
Layer 2: ContextBudgetAllocator             → ContextBudget
           allocate() — profile ratios → per-source token budgets
               │
Layer 3: EvidenceConstrainedContextSelector  → SelectionResult
           select() — requirement-first, dedup, marginal utility, overflow drop
               │
Layer 4: ContextBuilder                     → ContextPacket
           build() — source grouping, message construction, Qwen 3.5 compat
               │
       prompt_messages.py
           context_packet_to_model_messages() — validation gate
```

---

## Layer 0 — Model Profile (`model_profile.py`, 174 lines)

Resolves the effective context window. Only one model is registered:
`google/gemma-4-31B-it` (262 144 native, 1024 sliding window).

### Resolution order

```text
ResolvedContextWindow.effective_context_window =
  min(
    native_context_window,         # 262 144 for Gemma 4
    endpoint_context_window,       # from env vars
    application_context_cap         # APPLICATION_CONTEXT_CAP, default 262 144
  )
```

### Environment variables checked (first positive int wins)

| Variable | Purpose |
|---|---|
| `ENDPOINT_CONTEXT_WINDOW` | Most explicit override |
| `MODEL_CONTEXT_WINDOW` | OpenRouter-style override |
| `CONTEXT_LENGTH` | Generic framework name |
| `MAX_MODEL_LEN` | vLLM-style override |
| `MAX_INPUT_TOKENS` | Another framework name |

### Safe fallback

If no limits resolve: **4096 tokens** (`SAFE_FALLBACK_CONTEXT_WINDOW`). The
resolved window is traceable via `ResolvedContextWindow.limit_source` which
records which input was the binding constraint.

### Output

`ResolvedContextWindow` — frozen dataclass with model ID, native/endpoint/cap
values, effective window, limit source, and `to_metadata()` for trace inclusion.

## Layer 1 — Dynamic Budget (`dynamic_budget.py`, 183 lines)

`DynamicWorkingMemoryBudgetPlanner` resolves a route-specific working-memory cap
*before* per-source allocations. This is not the model's hard limit — it's a
softer "how much of the context window should working memory occupy" policy.

### Memory Budget Policy

| Route context | Budget cap | Rationale |
|---|---|---|
| `base` (general) | 4 096 | Lightweight chat |
| `memory_recall` | 8 192 | Retrieving structured memories |
| `chat` (single-chat recall) | 8 192 | Same-chat or cross-chat recall |
| `document` (single-scope) | 16 384 | Document QA |
| `multi_scope` | 16 384 | Multiple scope requirements |
| `long_document` | 32 768 | Document synthesis, summarization |
| `global_summary` | 65 536–131 072 | Full-history summarization |

### Required-evidence expansion

When the route plan demands specific evidence (e.g. `EXACT_QUOTE` requiring a
raw span) and the required evidence floor exceeds the normal route cap, the
budget expands past the cap — but never past the available memory budget (i.e.
the model's effective context window minus system overhead).

```text
working = min(
    available_memory_budget,   # context window — system tokens — safety
    route_cap,                 # from MemoryBudgetPolicy table above
    normal_target              # max(requested, required_floor + headroom)
)
```

A 25 % headroom ratio is added to the required evidence floor before comparison
(`required_evidence_headroom_ratio = 0.25`).

### Output

`DynamicBudgetPlan` — frozen dataclass with base budget, route cap, cap reason,
required evidence floor, headroom, available budget, working budget, and boolean
flags for `budget_expanded_for_required_evidence` and
`required_evidence_exceeds_available`.

## Layer 2 — Budget Allocation (`context_budget_allocator.py`, 337 lines)

`ContextBudgetAllocator` splits the working budget across memory sources using
five named allocation profiles.

### Five allocation profiles

| Profile | System | Recent | Structured | Document | Raw Span | Gists | Safety | Answer |
|---|---|---|---|---|---|---|---|---|
| `general_chat` | 8 % | 55 % | 20 % | 0 % | 0 % | 0 % | 7 % | 10 % |
| `memory_recall` | 8 % | 35 % | 35 % | 0 % | 0 % | 10 % | 7 % | 5 % |
| `document_question` | 8 % | 20 % | 10 % | 50 % | 0 % | 0 % | 7 % | 5 % |
| `mixed_memory_document` | 8 % | 20 % | 20 % | 25 % | 5 % | 10 % + 15 % | 7 % | 5 % |
| `global_summary` | 5 % | 5 % | 5 % | 10 % | 55 % | 20 % | 7 % | 5 % |

### Algorithm

1. **Subtract fixed costs**: `allocatable = context_limit − system_estimate − safety_margin − answer_reserve`
2. **Reserve minimums**: Each candidate-bearing source gets a floor (max 1, `minimum_candidate_source_tokens`, smallest candidate). If floor total exceeds allocatable, floors are scaled down equally.
3. **Normalized split**: Remaining tokens divided by enabled-source ratios (ratios renormalized to 1.0).
4. **Profile mapping**: `RoutePlan.context_profile` maps to allocator profile — `structured_memory_plus_recent_messages` → `memory_recall`; unknown profiles → `general_chat`.

### Output

`ContextBudget` — frozen dataclass with `max_tokens`, per-source `source_token_budgets: dict[str, int]`, `reserved_response_tokens`, and trace `metadata`.

## Layer 3 — Evidence Selection (`evidence_selector.py`, 1092 lines)

`EvidenceConstrainedContextSelector` — the most complex single file in the
project. Requirement-first deterministic selection of `MemoryCandidate` objects
under a token budget.

### Phase 1: Annotation

Every candidate is wrapped in a `CandidateAnnotation` recording trace ID,
source, rank, score, token cost, source message IDs, document/chunk IDs, parent
gist references, anchor message IDs, required scope matches, and an
`exact_raw_evidence` flag.

### Phase 2: Deduplication

- **Exact duplicates**: Content-normalized (whitespace-collapsed, lowercased).
  Merged candidate has retrieval paths unioned.
- **Overlapping spans**: Two raw-span candidates overlap if Jaccard on
  `source_message_ids > 0.7` and the candidate's anchors are fully covered by
  the keeper. The overlapping candidate is dropped.

### Phase 3: Required evidence (requirement-first)

Required evidence is selected before any optional candidates:

1. **EXACT_QUOTE** intent → requires a `raw_message_span` or `current_chat_span`.
2. **Required scopes** from route metadata → `document` (document_memory),
   `current_chat` (current_chat_span), `previous_chat` (raw_message_span, then
   previous_chat_gist fallback), `durable` (structured_memory).

For each requirement, the highest-ranked candidate from the preferred source is
selected. If it doesn't fit the global budget, it's dropped with
`global_budget_exceeded`. Missing requirements produce
`missing_required_evidence` records.

### Phase 4: Parent gist folding

When a `raw_message_span` candidate is selected and has a `parent_gist_id`, the
corresponding gist candidate (if already selected) is **folded** — removed from
selection, its tokens reclaimed. This avoids paying for both the lossy gist and
the exact span that supersedes it.

### Phase 5: Recent message suffix

The newest contiguous suffix of recent messages that fits the remaining budget
is selected. The latest user query is always excluded from the suffix (it's
appended separately as the final message).

### Phase 6: Optional candidates

Remaining candidates are scored by **marginal utility**:

```text
marginal_utility = relevance_score + source_prior + coverage_bonus − cost_penalty
```

Where:

- `relevance_score` = candidate's reranker score
- `source_prior` = per-source base weight (raw_message_span: 0.04, structured_memory: 0.025, etc.)
- `coverage_bonus` = 0.025 if source not yet represented
- `cost_penalty` = min(0.03, token_cost/total_budget × 0.03)

Candidates below `minimum_optional_utility` (0.15) terminate selection.

### Phase 7: Global summary mode

When `context_profile == "global_summary"`, a chronological coverage strategy
replaces marginal utility ranking. Gist candidates are selected first (oldest →
newest), then raw spans in timeline-coverage order (first, last, middle, then
max-spread fill). This ensures history coverage rather than relevance-driven
selection.

### Output

`SelectionResult` — mutable dataclass with `selected_candidates`,
`dropped_candidates` (with reasons), `evidence_contract_satisfied` boolean,
`missing_requirements`, `duplicate_decisions`, `selection_reasons`, per-candidate
`CandidateAnnotation` entries, utility scores, coverage metadata, and
`optional_selection_stopped_by` (reason the loop ended).

## Layer 4 — Context Building (`context_builder.py`, 627 lines)

`ContextBuilder` assembles a `ContextPacket` from ranked candidates and budget
information.

### Source grouping

Candidates are grouped by source in a fixed order:

```text
structured_memory → current_chat_gist → current_chat_span →
previous_chat_gist → document_memory → raw_message_span →
current_chat_chunks → previous_chat_memory → recent_messages
```

### Per-source budget enforcement

Each source gets its allocated `source_token_budget` from `ContextBudget`.
Candidates within that source are selected by rank order until the budget is
exhausted. Exceeding candidates are dropped with `source_budget_exceeded`.
Recent messages use a newest-contiguous-suffix strategy (picked from the tail
backward).

### Model message construction

Messages are built in OpenAI-compatible `[{"role": "...", "content": "..."}]` format:

1. **System message**: Base system prompt + merged structured memory
   (candidates formatted as `- category.key: content`)
2. **Recent messages**: Formatted as chat messages (role from metadata)
3. **Retrieved context**: Prepended to the latest user message as a
   `[Retrieved Context]` block with `[Question]` separator

### Qwen 3.5 compatibility

Qwen 3.5's chat template raises if any system message appears after the first.
The builder handles this by:

- Merging structured memory into the single leading system message
- Prepending all retrieved context (document chunks, gists, raw spans) to the
  **latest user message**, not as separate system messages

### Overflow handling

`drop_non_recent_candidates_for_overflow()` iteratively drops the lowest-ranked
non-recent candidate until the total prompt fits within the context limit. It
recomputes token accounting after each drop.

### Token accounting

`build_token_accounting()` produces per-section token counts: system tokens,
structured memory tokens, retrieved memory tokens (per source), recent message
tokens, latest user message tokens, total prompt tokens, answer reserve, safety
margin, and overflow detection.

### Output

`ContextPacket` — frozen dataclass with `system_prompt`, `structured_memory`,
`recent_message_ids`, `candidates`, `budget`, `model_messages`, and extensive
trace `metadata`.

## Supporting Files

### `prompt_messages.py` (110 lines) — Validation gate

`context_packet_to_model_messages()` validates model messages before they reach
the LLM:

| Validation | Failure reason |
|---|---|
| Context packet is `None` | `context_packet_missing` |
| No valid messages | `no_valid_messages` |
| First message is not system or empty | `system_prompt_missing` |
| Any message has invalid role | `invalid_message_role` |
| Any message has empty content | `empty_message_content` |
| Latest user message not found | `latest_user_message_missing` |
| Latest user message found more than once | `latest_user_message_duplicated` |
| Final message is not the user query | `latest_user_message_not_final` |
| Context comparison has severe warnings | `context_comparison_missing_latest_user_message` |

The latest user message may appear augmented (prepended context) — detection
uses substring matching with length checks.

### `context_comparator.py` (192 lines) — Legacy comparison

`ContextComparator` compares old `ShortTermMemory` context prompts against new
`ContextPacket` results. Used during the migration from the legacy context path
to the layered pipeline. Severe warnings (missing latest user message) block the
prompt switch.

### `token_estimator.py` (452 lines) — Token counting

Two implementations behind the `TokenEstimator` protocol:

- **`ApproximateTokenEstimator`**: 4 chars/token + per-message overhead. No
  dependencies. Used as fallback everywhere.
- **`GemmaTokenEstimator`**: Lazy-loads HuggingFace `AutoProcessor` and/or
  `AutoTokenizer` for `google/gemma-4-31B-it`. Falls back gracefully through
  processor → tokenizer → serialization → approximate on any failure.

Process-level caching (`_PROCESSOR_CACHE`, `_TOKENIZER_CACHE`) avoids reloading
assets. A thread lock protects concurrent access. Failures are cached so
subsequent calls don't retry.

## Design Principles

### Deterministic, not negotiated

Every layer is a deterministic function of its inputs. No LLM is consulted for
budgeting, selection, or message construction. The LLM only sees the final
assembled messages.

### Traceable

Every layer produces metadata that flows into the `ContextPacket` and,
ultimately, the `AnswerInspector`. Token accounting is per-source,
per-candidate, and per-section. Drop reasons are recorded with trace IDs.

### Requirement-first

Evidence requirements from the route plan (EXACT_QUOTE, required scopes) are
satisfied before any optional candidates are considered. This ensures the answer
agent always has the evidence the router deemed necessary.

### Qwen 3.5 constraints

The entire message construction is constrained by Qwen 3.5's single-system-message
requirement. All retrieved context is prepended to the user message, and
structured memory is merged into the system prompt.
