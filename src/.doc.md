# src/ — Root-Level Infrastructure

**TL;DR:** Eight root-level modules form the application foundation. `settings.py`
(271 lines) is the single source of truth — 67 constants pushed to `os.environ`.
`config.py` (197 lines) loads a frozen `AppConfig` from settings. `Database`
(1656 lines) manages 10 SQLite tables with stored dataclasses and 7 incremental
migrations. `ModelWrapper` (83 lines) wraps the OpenAI-compatible endpoint with a
`ConnectionGuard` pre-flight check. `ConnectionGuard` (189 lines) caches server
reachability with a configurable TTL. `ChatService` (477 lines) wires all agents
together. `api_routes.py` (381 lines) registers 21 FastAPI routes on Chainlit's
server. `chainlit_data_layer.py` (188 lines) adapts the project DB to Chainlit's
`BaseDataLayer`.

---

## Architecture: three-layer configuration

```
.env (secrets + overrides)
  │  load_dotenv()
  ▼
settings.py (67 constants → _setenv() → os.environ)
  │  AppConfig.load() reads module attributes + env
  ▼
config.py → AppConfig (frozen, injectable into all services)
```

`settings.py` is the single source of truth. All defaults live there as module-level
constants. `.env` overrides them. `config.py` freezes the resolved values into an
immutable `AppConfig` dataclass consumed by `ChatService` and `app.py`.

## `settings.py` — Canonical Configuration

### Purpose

Single source of truth for all configurable values. On import, `load_dotenv()` reads
`.env`, then every constant is pushed into `os.environ` via `_setenv(name, value)` —
but only when the env var isn't already set. This means `.env` wins over defaults.

### Startup mode logic

On import, `RERANKER_STARTUP_MODE` is checked and the reranker config is derived:

| Startup flag | `RERANKER_MODE` | CE model | CE weight | `RERANKER_HYBRID_BACKEND` |
|---|---|---|---|---|
| `hybrid` (default) | `hybrid` | `cross-encoder/ms-marco-MiniLM-L12-v2` | 0.65 | `cross_encoder` |
| `cross_encoder` | `cross_encoder` | `mixedbread-ai/mxbai-rerank-xsmall-v1` | 1.0 | `auto` |

This is how `startup.py --cross-encoder` vs `--hybrid` (the default) changes behavior.

## `config.py` — AppConfig

### Purpose

`AppConfig` is a frozen `@dataclass` (~60 fields) that centralizes all configuration.
Built by `AppConfig.load()` (aliased as `from_env()`). Reads directly from `src.settings`
module attributes plus a handful of env-only overrides. No module reads `os.getenv()`
directly after startup — all consumption goes through `AppConfig`.

### Factory

```python
config = AppConfig.from_env()
```

Resolves the endpoint context window and application cap from env vars (via
`model_profile.py`), then constructs one immutable instance. The dataclass is
`frozen=True` — no field can be mutated after construction.

### Fields include

**Model:** `openai_api_key`, `openai_base_url`, `model_name`,
`endpoint_context_window`, `endpoint_context_limit_source`, `application_context_cap`.

**Database:** `database_path` (from env or `data/chatbot.db`).

**All settings.py values** are mirrored: memory budgets, document config, reranker
config, routing config, gist config, context caps, debug flags.

**Important:** `RERANKER_CROSS_ENCODER_TOP_K=500` is a Python constant in
`reranker.py`, NOT an env var — no `settings.py` entry exists for it.

## `database.py` — SQLite Adapter

### Purpose

`Database` is the sole SQLite access layer — 1656 lines. All components that need
persistence go through it.

### Stored dataclasses (6 frozen types)

| Dataclass | Table | Key fields |
|---|---|---|
| `StoredChat` | `chats` | id, title, created_at, updated_at, model_name, active |
| `StoredMessage` | `messages` | id, chat_id, role, content, created_at, summarized, gist_processed |
| `StoredChatGist` | `chat_gists` | id, chat_id, source_type, gist_text, start_message_id, end_message_id |
| `StoredDocument` | `document_records` | id, file_name, status, source, chunk_count, error, summary_text |
| `StoredOperationResult` | `operation_results` | operation_id, operation_type, scope_id, result_ref |
| `StoredAnswerInspection` | `answer_inspections` | assistant_message_id, chat_id, trace_id, payload_json |

### 10 tables

| Table | Purpose |
|---|---|
| `chats` | Chat threads. `id TEXT PK`, `model_name`, `active INTEGER DEFAULT 1` |
| `messages` | User/assistant/system messages. `role CHECK(user/assistant/system)`, `summarized`, `gist_processed` |
| `chat_memory_state` | Per-chat LangMem snapshot cache. `chat_id PK FK`, `memory_json` |
| `chat_gists` | Gist summaries with source-message ranges. `source_type`, `start/end_message_id` |
| `long_term_memories` | Durable cross-chat structured memory. `UNIQUE(namespace_path, memory_id)` |
| `document_records` | Document lifecycle. `status CHECK(Uploading/Indexing/Ready/Failed/deleted)`, `summary_text` |
| `chat_documents` | Chat-document associations. `(chat_id, document_id) PK`, `selected` |
| `operation_results` | Idempotent operation keys for upload retry safety |
| `answer_inspections` | Per-answer observability. `assistant_message_id PK FK` |
| `document_summaries` | LLM-generated document summaries (populated by `DocumentIngestionAgent`) |

### Schema migrations (7 incremental fixes)

1. `_ensure_messages_summarized_column()` — adds `summarized` INTEGER
2. `_ensure_messages_gist_processed_column()` — adds `gist_processed` INTEGER + backfill
3. `_ensure_chats_model_name_column()` — adds `model_name` TEXT
4. `_ensure_chats_active_column()` — adds `active` INTEGER DEFAULT 1
5. `_drop_legacy_document_tables()` — drops `document_chunk_embeddings`, `document_chunks`, `documents`
6. `_ensure_document_summary_text_column()` — adds `summary_text` TEXT to `document_records`
7. `_ensure_document_status_check_allows_deleted()` — rebuilds table to widen CHECK

### Key operations

- **Fork:** Transactional copy — messages, gists, answer_inspections with full
  message-ID and chat-ID remapping. Does NOT copy `chat_memory_state` or
  `long_term_memories`. Forked messages marked `summarized=1`.
- **Documents:** `update_document_summary(document_id, summary_text)` stores LLM
  summaries. `document_summary(document_id)` retrieves them.
- **Idempotency:** `record_operation_once()` (INSERT OR IGNORE),
  `claim_document_upload()` (atomic operation-key claim + document-record creation).
- **Connection:** Uses `sqlite3.Row` row factory, PRAGMA foreign_keys=ON,
  busy_timeout=30s, WAL mode.

## `model_wrapper.py` — ModelWrapper

### Purpose

Thin wrapper (83 lines) around the `openai` SDK's `client.chat.completions.create()`.
Constructor accepts `config: AppConfig` and optional `model_name` override and
`connection_guard: ConnectionGuard`.

```python
def chat(self, messages: list[dict[str, str]], temperature: float | None = None) -> str
```

1. Calls `self._guard.check()` — fast-fails if server known-unreachable.
2. Sends to `/v1/chat/completions` with 35s timeout.
3. On `httpx.ConnectError` / `TimeoutException`: calls `guard.mark_unreachable()`,
   raises `InferenceServerUnreachable`.
4. Returns `completion.choices[0].message.content.strip()`.

No retry, no streaming. Error handling is the caller's responsibility (coordinator
catches `InferenceServerUnreachable` and `OpenAIError`).

## `connection_guard.py` — ConnectionGuard

### Purpose

Pre-flight health-check caching (189 lines). Prevents wasted LLM calls when the
inference server is down.

```python
class ConnectionGuard:
    def check(self) -> None      # Raises InferenceServerUnreachable if unhealthy
    def mark_unreachable(self, reason="") -> None
    def _ping(self) -> None      # GET /health → fallback GET base_url
```

### Caching

- Healthy cache: TTL `CONNECTION_GUARD_CACHE_TTL` (default 30s) — skips ping.
- Unhealthy cache: same TTL — fast-fails without ping.
- Ping: `GET {base_url}/health` → fallback `GET {base_url}`. 5xx → unhealthy.
  Timeout via `CONNECTION_GUARD_CHECK_TIMEOUT` (default 3s).

### Usage

`ModelWrapper.chat()` checks guard before every API call. `LangMemStructuredMemoryState`
checks before LangMem extraction. `connection_guard_from_env()` factory reads
`OPENAI_BASE_URL` from the environment.

## `chat_service.py` — ChatService

### Purpose

Top-level assembly point (477 lines). Constructs all agents, wires them together,
exposes the public API consumed by Chainlit and evals.

### Constructor (~35 parameters from AppConfig)

Wires:

- `ShortTermMemory` (database + model + memory-policy params)
- `CoordinatorAgent` (database, memory-agent, context-builder, chat-agent,
  retriever-dispatcher, routing-agent, memory-reranker, context-manager-agent)
- `DocumentIngestionAgent` (indexer + summary_model + summary_database)
- `PreviousChatGistGenerator` (lazy-constructed)
- `MemoryReranker` via `_build_reranker()` (mode-aware, preloads CE eagerly)

Shared `LangChainChromaRetriever` between ingestion and retrieval.

### Key methods

- **`start_chat(chat_id=None) -> str`** — creates chat row, optionally triggers
  previous-chat gist backfill.
- **`handle_user_turn(chat_id, content, **kw) -> AgentTurnResult`** — primary turn
  method. Guarded by `guarded_chat_operation()`. Validates active chat. Delegates
  to `coordinator.run_turn()`. Supports `defer_post_answer_memory_update` for
  deferred memory update after streaming response.
- **`handle_user_message(chat_id, content) -> str`** — convenience wrapper:
  turn → deferred memory → just the answer string.
- **`index_document_file(path, ...) -> DocumentFileIndexResult`** — full document
  upload: lock → create record → index → associate with chat → return.
- **`finalize_post_answer_memory_update(chat_id) -> bool`** — synchronous memory
  update after answer emission.

### System prompt

Module-level `SYSTEM_PROMPT` constant: evidence-constrained answers, conflict
resolution, "I don't know", no fabrication, source gaps → communicate them.

## `api_routes.py` — REST API Routes

### Purpose

Registers 21 FastAPI routes on Chainlit's internal server. Provides the CRUD surface
for Braemon and external tools.

### Registration

```python
register_api_routes(database, chat_service_getter)
```

Called from `app.py` after `ChatService` construction. Accesses `cl.server.app` and
prepends routes before Chainlit's catch-all.

### Route inventory

**Chats (9 routes):** `GET /api/chats`, `GET /api/chats/{id}/messages`, `POST /api/chats`,
`POST /api/chats/{id}/fork`, `POST /api/chats/{id}/end`,
`POST /api/chats/{id}/reactivate`, `POST /api/chats/{id}/consolidate`,
`GET /api/chats/{id}/consolidation-log`, `DELETE /api/chats/{id}`.

**Documents (5 routes):** `GET /api/documents`, `POST /api/documents/upload`,
`POST /api/documents/{id}/deactivate`, `POST /api/documents/{id}/activate`,
`DELETE /api/documents/{id}`.

**Memories (4 routes):** `GET /api/memories`, `POST /api/memories/{id}/deactivate`,
`POST /api/memories/{id}/activate`, `DELETE /api/memories/{id}`.

**System (3 routes):** `GET /api/models/status` (returns `{ready: bool}`),
`GET /api/retrieval-logs/{chat_id}/{turn_index}` (returns JSON log file),
`GET /api/stats` (counts: active chats, memories, ready documents; version `v2.4.1`).

## `chainlit_data_layer.py` — Chainlit Adapter

### Purpose

`SQLiteChainlitDataLayer` (188 lines) implements Chainlit's `BaseDataLayer`, translating
Chainlit's thread/step API into the project's `Database` operations.

### Key implementation

- **`get_thread(thread_id)`** — loads chat + messages via `messages_for_chat()`,
  converts to `ThreadDict`.
- **`list_threads(pagination, filters)`** — cursor-paginated `list_chats()` with search,
  returns `PaginatedResponse` with `hasNextPage`.
- **`update_thread(thread_id, name, ...)`** — title update or create-if-missing.
- **`delete_thread(thread_id)`** — cascade via FK.
- **`thread_display_name(chat)`** — appends `" · Ended"` to inactive chats.

Stub methods (feedback, element, step primitives) return no-ops — the project stores
everything in its own `messages` table and uses Chainlit only as WebSocket transport.
