# src/ — Root-Level Infrastructure

**TL;DR:** Six root-level modules form the application foundation. `AppConfig` (297 lines) loads
~90 env-backed frozen fields. `Database` (1598 lines) manages 9 SQLite tables with stored
dataclasses, schema migrations, and fork logic. `ModelWrapper` (59 lines) wraps the
OpenAI-compatible `/v1/chat/completions` endpoint. `ChatService` (431 lines) wires all agents
together and exposes `handle_user_turn()`. `api_routes.py` (340 lines) registers 16 FastAPI
routes on Chainlit's server. `chainlit_data_layer.py` (188 lines) adapts the project DB to
Chainlit's `BaseDataLayer`.

## `config.py` — AppConfig

### Purpose

`AppConfig` is a frozen `@dataclass` that centralizes all environment-variable configuration.
Every tunable in the system is loaded via `AppConfig.from_env()` and passed through
constructor injection down to agents, retrievers, and services. No module reads `os.getenv()`
directly after startup.

### Factory: `from_env()`

```python
config = AppConfig.from_env()
```

1. Calls `load_dotenv()` to pull in `.env` file overrides.
2. Resolves the endpoint context window via `endpoint_context_limit_from_env()` and the
   application cap via `application_context_cap_from_env()` (imported from `model_profile.py`).
3. Constructs one `AppConfig` instance with ~90 fields, each falling back to a hardcoded
   default when the env var is unset.

The dataclass is `frozen=True` — no field can be mutated after construction.

## `database.py` — SQLite Adapter

### Purpose

`Database` is the sole SQLite access layer. All components that need persistence go
through this class. It handles schema creation, migrations, CRUD operations, and the
transactional fork operation.

### Stored dataclasses (6 frozen types)

| Dataclass | Table mapped | Fields |
|---|---|---|
| `StoredChat` | `chats` | id, title, created_at, updated_at, model_name, active |
| `StoredMessage` | `messages` | id, chat_id, role, content, created_at, summarized, gist_processed |
| `StoredChatGist` | `chat_gists` | id, chat_id, source_type, gist_text, topics_json, decisions_json, open_tasks_json, start_message_id, end_message_id, created_at, updated_at, metadata_json |
| `StoredDocument` | `document_records` | id, file_name, status, source, chunk_count, error, created_at, updated_at, metadata_json |
| `StoredOperationResult` | `operation_results` | operation_id, operation_type, scope_id, result_ref, created_at |
| `StoredAnswerInspection` | `answer_inspections` | assistant_message_id, chat_id, trace_id, payload_json, created_at |

### 9 tables

| Table | Purpose | Key columns |
|---|---|---|
| `chats` | Chat threads | id TEXT PK, title, model_name, active INTEGER |
| `messages` | Raw user/assistant/system messages | id AUTO PK, chat_id FK, role CHECK, summarized, gist_processed |
| `chat_memory_state` | Per-chat LangMem snapshot cache | chat_id PK FK, memory_json |
| `chat_gists` | Gist summaries with source-message ranges | id PK, chat_id FK, source_type, start_message_id, end_message_id |
| `long_term_memories` | Durable cross-chat structured memory | id PK, namespace_path, memory_id, category, key, value, status UNIQUE(namespace_path, memory_id) |
| `document_records` | Document lifecycle metadata | id TEXT PK, status CHECK(Uploading/Indexing/Ready/Failed/deleted) |
| `chat_documents` | Chat-document associations | (chat_id, document_id) PK, selected |
| `operation_results` | Idempotent operation keys | operation_id PK, operation_type, scope_id, result_ref |
| `answer_inspections` | Per-answer observability | assistant_message_id PK FK, chat_id, trace_id, payload_json |

### Schema migrations (7 incremental fixes)

`init_schema()` always runs the base `CREATE TABLE IF NOT EXISTS` script, then applies
incremental fixes:

1. **`_ensure_messages_summarized_column()`** — adds `summarized` INTEGER column for
   databases created before Short-Term Memory v2.
2. **`_ensure_messages_gist_processed_column()`** — adds `gist_processed` INTEGER column.
   Backfills existing gist-processed messages by matching `chat_gists` message-id ranges.
3. **`_ensure_chats_model_name_column()`** — adds `model_name` TEXT column.
4. **`_ensure_chats_active_column()`** — adds `active` INTEGER (default 1).
5. **`_drop_legacy_document_tables()`** — drops abandoned `document_chunk_embeddings`,
   `document_chunks`, `documents` tables within a SAVEPOINT.
6. **`_ensure_document_status_check_allows_deleted()`** — rebuilds the
   `document_records` table to widen the status CHECK constraint to include `deleted`.
   Uses the create-insert-drop-rename pattern (SQLite can't ALTER CHECK constraints).
7. Creates two composite indexes: `idx_messages_chat_summarized` and
   `idx_messages_chat_gist_processed`.

### Connection management

```python
with self.connect() as connection:
    # connection.row_factory = sqlite3.Row
    # PRAGMA foreign_keys = ON
    # auto-commit on success, auto-close on exit
```

Uses `sqlite3.Row` row factory (dict-like access via key or index). Foreign keys are
enforced. WAL mode is not set explicitly — relies on the default (which is WAL on
most platforms since SQLite 3.37).

### Key operations

**Chat lifecycle:** `create_chat()` (INSERT OR IGNORE), `get_chat()`, `list_chats()`
(cursor-based pagination with search), `update_chat_title()`, `mark_chat_active()`,
`mark_chat_inactive()`, `delete_chat()` (cascade-deletes messages, gists, memory-state).

**Fork (`fork_chat()`):** Transactional copy of a chat into a new ID. Copies messages
(remapped to new IDs via `message_id_map`), chat gists (remapped message-id anchors
and chat-ID provenance), and answer inspections (remapped through the same maps).
**Does not copy** `chat_memory_state` (per-chat cache) or namespaced long-term memories
(shared across chats). Forked messages are marked `summarized=1` so only post-fork
messages can produce new global structured memories. Uses `remap_message_id()` and
`remap_chat_local_json()` for recursive provenance remapping across 3 metadata key sets
(`MESSAGE_ID_METADATA_KEYS`, `MESSAGE_ID_LIST_METADATA_KEYS`, `CHAT_ID_METADATA_KEYS`).

**Messages:** `save_message()` (returns `lastrowid`), `recent_messages()` (reverse
chronological, limited), `messages_for_chat()` (all, chronological), `messages_for_chat_span()`
(inclusive message-id range), `old_unsummarized_messages()` (outside the raw recent window),
`old_ungisted_messages()` (gist-unprocessed outside recent window).

**Document lifecycle:** `create_document_record()`, `update_document_status()`,
`associate_document_with_chat()`, `get_document()`, `delete_document()` (soft: status →
deleted), `hard_delete_document()` (cascade via FK), `documents_for_chat()` (JOIN query
with status filters).

**Idempotent operations:** `record_operation_once()` (INSERT OR IGNORE, returns True if
new), `claim_document_upload()` (atomic operation-key claim + document-record creation).
Returns `(claimed, document_id)` — when already claimed, verifies same scope/type and
returns the existing document ID.

**Memory state:** `chat_memory_state()`, `upsert_chat_memory_state()`,
`mark_messages_summarized()`, `mark_messages_gist_processed()`, `insert_chat_gist()`
(transactional gist + message-state advancement).

### Design decisions

- **Auto-migration at startup:** `init_schema()` is called in `__init__()`. No external
  migration tool or manual migration step needed.
- **SAVEPOINT for dangerous changes:** The legacy table drop uses a SAVEPOINT so a
  crash mid-migration doesn't leave a broken schema.
- **Fork provenance remapping:** The `fork_chat()` method doesn't just copy rows — it
  recursively walks JSON metadata fields to rewrite every message-ID and chat-ID
  reference. This is critical because cross-chat retrievers use these provenance
  markers to find evidence.
- **Soft-delete pattern:** Documents use `status='deleted'` rather than actual deletion.
  `list_all_documents()` defaults to excluding deleted records. Chat-active is
  `active INTEGER` with `mark_chat_inactive/active()`.

## `model_wrapper.py` — ModelWrapper

### Purpose

Thin wrapper around the `openai` SDK's `client.chat.completions.create()`. A single
`ModelWrapper` instance is constructed at startup and shared across all agents that
need LLM calls (chat, LangMem, LLM reranker, gist extractor).

### Construction

```python
wrapper = ModelWrapper(AppConfig.from_env())
# or with explicit model name override:
wrapper = ModelWrapper(config, model_name="qwen2.5:3b")
```

Field `self.model_name` defaults to `config.model_name` (`"google/gemma-4-31B-it"`).
The OpenAI client is configured with `max_retries=0` (no automatic retry) and a
unified httpx timeout of `MODEL_REQUEST_TIMEOUT` (default 35s) across connect,
read, write, and pool.

### `chat()` method

```python
def chat(self, messages: list[dict[str, str]], temperature: float | None = None) -> str:
```

- Sends messages to `self.model_name` via `/v1/chat/completions`.
- Optional `temperature` parameter (passed through only when not None).
- Returns `completion.choices[0].message.content` stripped of whitespace.
- On empty content, returns `""`.

No retry logic, no streaming support, no response parsing beyond extracting
`.message.content`. Error handling (connection errors, timeouts, API errors) is left
to callers.

### Why it exists as a separate class

Other repos often inline the OpenAI client call. The wrapper exists to:

1. Own the model identity — agents don't know which model they're calling.
2. Centralize timeout and retry configuration.
3. Provide a single mock point for tests.
4. Keep the OpenAI dependency at one boundary.

## `chat_service.py` — ChatService

### Purpose

`ChatService` is the **top-level assembly point**. It constructs all agents (coordinator,
retrievers, reranker, context manager, memory, previous-chat-gist generator) and exposes
the public-turn and document-indexing APIs consumed by Chainlit and the evals harness.

### Construction

`ChatService.__init__()` accepts ~35 parameters (all from `AppConfig`) and wires:

- `ShortTermMemory` (from `database`, `model`, and memory-policy params)
- `CoordinatorAgent` (from database, memory-agent, context-builder, chat-agent,
  retriever-dispatcher, routing-agent, memory-reranker, context-manager-agent)
- `DocumentIngestionAgent` (from indexer)
- `PreviousChatGistGenerator` (constructed lazily in `start_chat()` or `build_previous_chat_gist_generator()`)

No dependency injection framework — all wiring is manual constructor injection in `__init__()`.

### Key public methods

#### `start_chat(chat_id=None) -> str`

Creates a chat row with `create_chat()`, sets title to `"Chainlit chat"`, records
model name. If `previous_chat_gist_generation_enabled`, calls
`generator.generate_for_existing_chats(active_chat_id=chat_id)` to gist any
finished chats without gists.

#### `handle_user_turn(chat_id, content, **kw) -> AgentTurnResult`

The **primary turn method**. Wraps the call in `guarded_chat_operation()` (active-check
lock from `operation_guard.py`). Validates the chat is active. Optionally sets title
from first message. Delegates to `coordinator.run_turn()`. Calls
`persist_answer_inspection()` after the turn.

Parameters:
- `orchestration_mode` — forwarded to coordinator for LangGraph shadow/demo paths.
- `task_context` — forwarded to coordinator (e.g. `"document_qa"`).
- `persisted_user_message_id` — when non-None, validates message matches content.
- `defer_post_answer_memory_update` — when True, memory update is deferred to a
  separate `finalize_post_answer_memory_update()` call.

#### `handle_user_message(chat_id, content) -> str`

Convenience wrapper: calls `handle_user_turn(defer_post_answer_memory_update=True)`
then `finalize_post_answer_memory_update()`. Returns just the answer string (used by
Chainlit's `on_message` callback).

#### `index_document_file(path, display_name, *, chat_id, operation_id) -> DocumentFileIndexResult`

Full document upload lifecycle:

1. Enters `guarded_chat_operation()` lock.
2. Checks for prior `operation_id` to support idempotent retry.
3. Creates document record via `create_document_record()` or `claim_document_upload()`.
4. Transitions status: `Uploading → Indexing → Ready` (or `Failed` on error).
5. Calls `document_ingestion_agent.index_file()` for Chroma indexing.
6. Associates document with chat via `associate_document_with_chat()`.
7. Returns `DocumentFileIndexResult(file_name, document_id, chunk_count)`.

#### `finalize_post_answer_memory_update(chat_id) -> bool`

Runs `self.memory.update_memory_if_needed()` synchronously after the answer has been
emitted. Catches `OpenAIError` silently (returns False). Used in the `handle_user_message`
flow where memory update is deferred to after the streaming response.

### Previous-chat gist generator construction

`build_previous_chat_gist_generator()`:

- If extractor is `"llm"`: uses `FallbackChatGistExtractor(LLMChatGistExtractor, DeterministicPreviousChatGistExtractor)`.
- Otherwise: uses `DeterministicPreviousChatGistExtractor` directly.
- Wraps in `PreviousChatGistGenerator` with the configured `max_messages_per_gist`.

### System prompt

`SYSTEM_PROMPT` is a module-level constant instructing the model to be concise, use
contextual evidence, not invent facts, and state conflicts or limitations explicitly.

## `api_routes.py` — REST API Routes

### Purpose

Registers 16 FastAPI routes on Chainlit's internal FastAPI server. Provides the
CRUD surface that Braemon (and any external tool) calls via `/api/*`.

### Registration

```python
register_api_routes(database, chat_service_getter)
```

Called from `app.py` after `ChatService` is constructed. Stores database and a
`lambda: chat_service` getter in module-level globals. Accesses Chainlit's server
via `cl.server.app` and prepends routes before Chainlit's catch-all.

### Route table

#### Chats (8 routes)

| Method | Path | Purpose |
|---|---|---|
| `GET` | `/api/chats` | List chats with `limit`, `cursor`, `search`; requires messages |
| `GET` | `/api/chats/{chat_id}/messages` | Message history for a chat |
| `POST` | `/api/chats` | Create new chat via `svc.start_chat()` |
| `POST` | `/api/chats/{chat_id}/fork` | Fork chat to new UUID |
| `POST` | `/api/chats/{chat_id}/end` | Mark chat inactive |
| `POST` | `/api/chats/{chat_id}/reactivate` | Mark chat active |
| `POST` | `/api/chats/{chat_id}/consolidate` | Force memory processing with 30s timeout |
| `DELETE` | `/api/chats/{chat_id}` | Hard delete (cascade via FK) |

`consolidate` runs `memory.process_all_for_chat_end()` in `asyncio.to_thread()` with
a 30-second timeout. On timeout, returns 504.

#### Documents (5 routes)

| Method | Path | Purpose |
|---|---|---|
| `GET` | `/api/documents` | List documents with `limit` and `status` filter |
| `POST` | `/api/documents/upload` | Upload file via multipart/form-data, index globally |
| `POST` | `/api/documents/{doc_id}/deactivate` | Soft-delete (status → deleted) |
| `POST` | `/api/documents/{doc_id}/activate` | Restore (status → Ready) |
| `DELETE` | `/api/documents/{doc_id}` | Hard delete + Chroma chunk cleanup via `delete_document_chunks()` |

The upload route reads multipart form data from `cl.context.request`, writes to a
temp file, calls `svc.index_document_file()`, and cleans up the temp file.

#### Memories (4 routes)

| Method | Path | Purpose |
|---|---|---|
| `GET` | `/api/memories` | List memories with `limit` and `status` filter |
| `POST` | `/api/memories/{memory_id}/deactivate` | Soft-delete (status → deleted) |
| `POST` | `/api/memories/{memory_id}/activate` | Restore (status → active) |
| `DELETE` | `/api/memories/{memory_id}` | Hard delete from long_term_memories |

Memory deactivate/activate/delete operate directly on `long_term_memories` table via
`memory_id`. Note: `memory_id` is the LangMem-specific ID, not `id` (the SQLite row
PK). Hard delete uses `DELETE FROM long_term_memories WHERE memory_id = ?`.

#### System (1 route)

| Method | Path | Purpose |
|---|---|---|
| `GET` | `/api/stats` | Counts: active chats, active memories, ready documents; returns version `"v2.4.1"` |

### Error handling

Every route wraps its logic in `try/except Exception` returning `{"error": str(e)}, 500`.
The document upload route also raises `fastapi.HTTPException` for content-type and
file-missing errors (which FastAPI converts to proper 400 responses).

### Route ordering

After registration, new routes are moved to the front of `app.router.routes`:

```python
new_routes = app.router.routes[initial_route_count:]
app.router.routes = new_routes + app.router.routes[:initial_route_count]
```

This ensures `/api/chats`, `/api/documents`, etc. take priority over Chainlit's catch-all.

## `chainlit_data_layer.py` — Chainlit Adapter

### Purpose

`SQLiteChainlitDataLayer` implements Chainlit's `BaseDataLayer` abstract class,
translating Chainlit's thread/step/element API into the project's SQLite `Database`
operations. This is what makes the project's custom DB schema work with Chainlit's
built-in UI components.

### Class: `SQLiteChainlitDataLayer`

```python
class SQLiteChainlitDataLayer(BaseDataLayer):
    def __init__(self, database: Database) -> None:
        self.database = database
```

#### Implemented Chainlit methods

| Method | Implementation | Notes |
|---|---|---|
| `get_user(identifier)` | Returns `persisted_user(identifier)` | Stable local user |
| `create_user(user)` | Returns `persisted_user(user.identifier)` | No-op (single-user) |
| `get_thread(thread_id)` | Loads chat + all messages via `messages_for_chat()` | Converts to `ThreadDict` with `steps` |
| `list_threads(pagination, filters)` | Cursor-paginated `list_chats()` with search | Returns `PaginatedResponse` with `hasNextPage` |
| `update_thread(thread_id, name, ...)` | Title update or create-if-missing | Uses `update_chat_title()` / `create_chat()` |
| `delete_thread(thread_id)` | `database.delete_chat(thread_id)` | Cascade via FK |

#### Unsupported / stub methods

The following `BaseDataLayer` methods return no-ops or empty values because the
project doesn't use Chainlit's feedback, element, or step primitives:

`delete_feedback()`, `upsert_feedback()`, `create_element()`, `get_element()`,
`delete_element()`, `create_step()`, `update_step()`, `delete_step()`,
`get_favorite_steps()`, `build_debug_url()`, `close()`, `get_thread_author()`.

This is intentional — the project stores everything in its own `messages` table
and uses Chainlit only as a WebSocket transport and session manager.

### Conversion helpers

- **`persisted_user(identifier)`** — returns a fixed `PersistedUser` with id
  `"local-user"`, hardcoded display name `"Local user"`, and epoch creation date.
- **`thread_from_chat(chat, steps)`** — maps `StoredChat` → `ThreadDict`. Embeds
  `model_name` and `active` into metadata. Inactive chats get `status: "Ended"`.
- **`thread_display_name(chat)`** — appends `" · Ended"` suffix to inactive chat
  titles without changing the persisted title.
- **`step_from_message(message, thread_id)`** — maps `StoredMessage` → `StepDict`.
  Maps roles to Chainlit step types: `user` → `user_message`, `assistant` →
  `assistant_message`, `system` → `system_message`. Embeds `message_id` and
  `summarized` in step metadata for traceability.
