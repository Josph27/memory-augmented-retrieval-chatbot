**TL;DR:** Read-only per-answer observability layer. `build_answer_inspection()` serializes the full `AgentTurnResult` trace into a bounded JSON payload. `persist_answer_inspection()` saves it to SQLite as a best-effort side effect â€” failures never block the answer. `inspection_rows_for_ui()` loads and joins inspections with their assistant message text for the frontend "Inspect answer" feature.

---

## Purpose

The inspection subsystem provides post-hoc answer observability without affecting the hot path. It captures routing decisions, evidence provenance, token diagnostics, and orchestration metadata in a single JSON payload stored alongside the assistant message.

**Key design constraint:** persistence is best-effort. If `persist_answer_inspection()` throws, the error is swallowed â€” the user's answer is never withheld because of an inspection write failure.

## Core functions

### `build_answer_inspection(result, database) â†’ dict`

Serializes the full `AgentTurnResult` into a bounded read-only JSON payload. Never mutates state. Returns a version-1 dict with top-level keys: `overview`, `evidence_summary`, `selected_sources`, `retrieval_diagnostics`, `documents`.

**Payload structure (version 1):**

```json
{
  "version": 1,
  "assistant_message_id": 42,
  "chat_id": "abc123",
  "trace_id": "uuid-...",
  "overview": {
    "requested_mode": "langgraph_demo",
    "effective_mode": "langgraph_demo",
    "authoritative_context": "langgraph",
    "graph_executed": true,
    "native_fallback_used": false,
    "route": ["recent_messages", "structured_memory", "document_memory"],
    "route_intent": "document_question",
    "context_profile": "document_question"
  },
  "evidence_summary": {
    "retrieved_candidate_count": 15,
    "reranked_candidate_count": 15,
    "selected_evidence_count": 6,
    "selected_context_tokens": 4200,
    "final_prompt_tokens": 5200,
    "evidence_validation": true
  },
  "selected_sources": [ /* candidate rows â€” see below */ ],
  "retrieval_diagnostics": {
    "document_fallback_used": false,
    "retrieval_errors": [],
    "evidence_validation": true,
    "dropped_candidate_count": 3,
    "dropped_candidates": [ /* drop reason rows */ ]
  },
  "documents": [ /* document summary rows */ ]
}
```

### `persist_answer_inspection(result, database) â†’ bool`

Best-effort wrapper. Calls `build_answer_inspection()` then `database.save_answer_inspection()`. Returns `False` silently on any exception â€” callers should never gate answer delivery on this result.

### `inspection_rows_for_ui(database, chat_id) â†’ list[dict]`

Loads all inspection payloads for a chat and joins them with their assistant message text. Each payload is JSON-loaded, validated, and augmented with:

- `assistant_message_id`: the FK back to messages
- `answer_text`: the full `message.content` from the assistant row

Invalid JSON rows are silently skipped. Only rows with a matching assistant message are included.

## Bounding and safety

All data is bounded to prevent oversized payloads:

| Constant | Value | Applied to |
|---|---|---|
| `MAX_EXCERPT_CHARS` | 280 | Candidate content excerpts |
| `MAX_SELECTED_ITEMS` | 24 | Selected candidates in `selected_sources` |
| `MAX_DROPPED_ITEMS` | 24 | Dropped/skipped candidates in diagnostics |
| Per-string limit | 240 | `retrieval_paths` and other string lists |

## What's included vs. excluded

**Included:**

- Routing overview (mode, sources, intent, profile)
- Evidence summary (candidate counts, token usage, validation)
- Per-selected-candidate provenance (source, score, rank, document/chunk IDs, message ranges, timestamp, source_range)
- Retrieval diagnostics (fallback usage, errors, dropped candidates with reasons, overlap info)
- Document metadata (filename, status, chunk count, whether selected as evidence)

**Excluded:**

- Full candidate text (only 280-char excerpts)
- Full prompt text
- LangMem extraction details
- Answer generation internals beyond errors
- System prompt content
