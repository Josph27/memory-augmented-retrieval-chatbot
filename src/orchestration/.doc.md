## Why Dual Orchestration Exists

The LangGraph pipeline is an experimental read-only comparison point. It
re-implements the full retrieval â†’ reranking â†’ context pipeline as a
`StateGraph` so the team can:

- Compare source selection and token allocations between implementations.
- Validate evidence contracts declaratively.
- Profile node-level timings independently.
- Test the Semantic Router v2 in isolation.

The Native path is always safe and always runs. LangGraph may run alongside
or override Native, but never replaces the write path.

## Three Modes

| Mode | Constant | Authoritative Packet | LangGraph Runs? |
|------|----------|---------------------|-----------------|
| Native | `NATIVE` | Native | No |
| LangGraph Shadow | `LANGGRAPH_SHADOW` | Native | Yes (read-only comparison) |
| LangGraph Demo | `LANGGRAPH_DEMO` | LangGraph (fallback: Native) | Yes |

**Native** (`"native"`):

- Only the `CoordinatorAgent` imperative pipeline runs.
- No LangGraph graph is built or invoked.
- This is the production-safe default.

**LangGraph Shadow** (`"langgraph_shadow"`):

- Native runs first and remains authoritative.
- LangGraph runs separately for comparison.
- An `OrchestrationComparison` is produced and attached to the trace.
- If the graph fails, Native still delivers the answer â€” no fallback needed.

**LangGraph Demo** (`"langgraph_demo"`):

- Native runs first (for preparation, memory update triggers, ChatAgent).
- LangGraph runs and its `ContextPacket` becomes authoritative.
- If the graph fails or produces an invalid packet, Native is used as fallback.
- `orchestration_fallback_used` is set to `True` on fallback.
- This is a demo mode â€” not guaranteed for production correctness.

Mode normalization (`normalize_orchestration_mode`):

- Strips and lowercases the input.
- Unknown values fall back to `NATIVE`.
- Only `"native"`, `"langgraph_shadow"`, `"langgraph_demo"` are valid.

## LangGraph Pipeline Architecture

The graph is built by `build_langgraph_memory_pipeline()` from
`langgraph_memory_pipeline.py`. It is a read-only `StateGraph` with state
typed as `MemoryGraphState` (a `TypedDict`).

### Node Sequence

```
START â†’ route â†’ retrieve â†’ expand_gists â†’ rerank â†’ build_context
                                                          â†“
                                                  validate_evidence
                                                    â†™            â†˜
                                          mock_answer          insufficient_evidence
                                                    â†˜            â†™
                                                      trace â†’ END
```

### Node Details

**1. `route`** â€” Routing

- By default uses `SemanticRouter` (not `RoutingAgent`).
- Produces `RoutePlan`, `SemanticRoutePlan`, and `EvidenceContract`.
- Attaches routing metadata for trace observability.

**2. `retrieve`** â€” Candidate Retrieval

- Iterates enabled sources from the `RoutePlan`.
- Dispatches to individual retrievers via `RetrieverDispatcher`.
- Caps results at `MAX_DIRECT_RETRIEVAL_CANDIDATES` (160).
- Errors per-source are collected, not fatal.

**3. `expand_gists`** â€” Gist Expansion

- Runs `GistRawSpanExpander` on retrieved candidates.
- Expands gist-orientation candidates into bounded raw evidence.
- Caps expanded candidates at `MAX_GIST_EXPANSION_CANDIDATES` (80).

**4. `rerank`** â€” Reranking

- Merges base and expanded candidates.
- Uses `MemoryReranker.rank_with_trace()`.
- Applies the `ranking_profile` from the `RoutePlan`.
- Query for reranking comes from `retrieval_query_for_reranking()`.

**5. `build_context`** â€” Context Construction

- Delegates to `ContextManagerAgent.build_context_packet()`.
- Produces the `ContextPacket` with budget allocations and selected evidence.
- Records `source_budgets` and per-source token usage.

**6. `validate_evidence`** â€” Evidence Contract Validation

- Checks the `EvidenceContract` against sources present in the `ContextPacket`.
- Four failure conditions:
  - Raw span required but absent.
  - Document citation required but absent.
  - Structured memory required but absent.
  - Only gist sources present when `must_not_answer_from_gist_only` is set.
- Conditional branch to `mock_answer` or `insufficient_evidence`.

**7. `mock_answer` / `insufficient_evidence`** â€” Terminal

- `mock_answer`: produces `"MOCK ANSWER: evidence contract satisfied â€¦"`.
- `insufficient_evidence`: produces `"MOCK INSUFFICIENT EVIDENCE: â€¦"`.
- Neither node calls an LLM or generates real answers.

**8. `trace`** â€” Observability

- Collects route sources, candidate identities, source counts, dropped
  candidates, budgets, token usage, provenance validity, contract state,
  errors, and node-level timings.
- Candidate traces are bounded (max 20, snippets â‰¤ 160 chars).

### Key Differences from Native

| Aspect | Native | LangGraph |
|--------|--------|-----------|
| Router | `RoutingAgent` (rule/LLM/hybrid) | `SemanticRouter` by default |
| Gist expansion | Implicit in retrieval | Explicit `expand_gists` node |
| Evidence validation | In context builder | Explicit `validate_evidence` node with contract |
| Answer generation | `ChatAgent` (real LLM) | Mock strings only |
| Database writes | Via `ShortTermMemoryAgent` | None |

## Write-Safety: Graph Is Read-Only

The LangGraph graph **never writes to the database**. It has no access to
`ShortTermMemoryAgent`, `LangMemStructuredMemoryState`, `LongTermStore`, or
any write path. It only reads through retrievers and produces a
`ContextPacket`.

All memory updates, message persistence, and gist generation happen through
the Native path â€” which always runs first, even in `LANGGRAPH_DEMO` mode.

The graph is intentionally incomplete for production:

- No real LLM answer generation.
- No database writes.
- No chat lifecycle actions (end, fork).
- No document ingestion.

---

## Dual-Path Safety: Native Always Runs First

In `coordinator_agent.py`, the orchestration sequence is:

1. **Native preparation always runs first.** The `CoordinatorAgent` builds
   the Native `OrchestrationResult` unconditionally.
2. If mode is not `NATIVE`, the LangGraph graph is invoked.
3. In `LANGGRAPH_DEMO` mode, the LangGraph result replaces Native as
   authoritative. If the graph fails, Native is the fallback.
4. Legacy context building (ShortTermMemoryAgent) still runs against Native.
5. The `ChatAgent` still uses Native infrastructure for answer generation.

This guarantees that even if the LangGraph graph is broken, the system
degrades gracefully to the Native answer.

## OrchestrationComparison

The `OrchestrationComparison` frozen dataclass captures bounded differences
between Native and LangGraph results without exposing content:

| Field | Meaning |
|-------|---------|
| `native_sources` | Source types in Native `ContextPacket` |
| `langgraph_sources` | Source types in LangGraph `ContextPacket` |
| `native_only_sources` | Sources only in Native |
| `langgraph_only_sources` | Sources only in LangGraph |
| `selected_candidate_overlap` | Count of shared (source, record_id) pairs |
| `native_selected_count` | Candidate count in Native |
| `langgraph_selected_count` | Candidate count in LangGraph |
| `native_prompt_tokens` | Estimated prompt tokens in Native |
| `langgraph_prompt_tokens` | Estimated prompt tokens in LangGraph |
| `token_difference` | LangGraph tokens minus Native tokens |

Used in `LANGGRAPH_SHADOW` mode for diagnostics. Never exposes candidate text
or transcript content.
