# Orchestration — `.doc.md`

**TL;DR:** Two parallel orchestration implementations: a production Native imperative
path and a LangGraph read-only graph. Three modes control which is authoritative
(default: `"native"`). The 10-node graph runs route → retrieve → expand_gists →
rerank → expand_doc_neighbors → build_context → validate_evidence →
(mock_answer | insufficient_evidence) → trace. It is read-only — all writes go
through Native.

---

## Why Dual Orchestration Exists

The LangGraph pipeline is an experimental read-only comparison point. It
re-implements the full retrieval → reranking → context pipeline as a
`StateGraph` so the team can:

- Compare source selection and token allocations between implementations.
- Validate evidence contracts declaratively.
- Profile node-level timings independently.
- Test the Semantic Router v2 in isolation.

The Native path is always safe and always runs. LangGraph may run alongside
or override Native, but never replaces the write path.

---

## Three Modes

| Mode | Constant | Authoritative Packet | LangGraph Runs? |
|------|----------|---------------------|-----------------|
| Native | `NATIVE` | Native | No |
| LangGraph Shadow | `LANGGRAPH_SHADOW` | Native | Yes (read-only comparison) |
| LangGraph Demo | `LANGGRAPH_DEMO` | LangGraph (fallback: Native) | Yes |

**Native** (`"native"`):

- Only the `CoordinatorAgent` imperative pipeline runs.
- No LangGraph graph is built or invoked.
- This is the production-safe default.

**LangGraph Shadow** (`"langgraph_shadow"`):

- Native runs first and remains authoritative.
- LangGraph runs separately for comparison.
- An `OrchestrationComparison` is produced and attached to the trace.
- If the graph fails, Native still delivers the answer — no fallback needed.

**LangGraph Demo** (`"langgraph_demo"`):

- Native runs first (for preparation, memory update triggers, ChatAgent).
- LangGraph runs and its `ContextPacket` becomes authoritative.
- If the graph fails or produces an invalid packet, Native is used as fallback.
- `orchestration_fallback_used` is set to `True` on fallback.
- This is a demo mode — not guaranteed for production correctness.

Mode normalization (`normalize_orchestration_mode`):

- Strips and lowercases the input.
- Unknown values fall back to `NATIVE`.
- Only `"native"`, `"langgraph_shadow"`, `"langgraph_demo"` are valid.

## LangGraph Pipeline Architecture

The graph is built by `build_langgraph_memory_pipeline()` from
`langgraph_memory_pipeline.py`. It is a read-only `StateGraph` with state
typed as `MemoryGraphState` (a `TypedDict`).

### Services Wiring

`LangGraphSpikeServices` bundles existing typed-memory services:

| Field | Purpose |
|-------|---------|
| `routing_agent` | `RoutingAgent` for rule/LLM/hybrid routing (optional when Semantic Router enabled) |
| `semantic_router` | `SemanticRouter` v2 for declarative routing (default) |
| `use_semantic_router` | Toggle routing backend |
| `dispatcher` | `RetrieverDispatcher` for source-specific retrieval |
| `reranker` | `MemoryReranker` (defaults to deterministic if not injected) |
| `context_manager` | `ContextManagerAgent` for budget + evidence + packet assembly |
| `system_prompt` | Injected system prompt text |

### 10-Node Sequence

```
START → route → retrieve → expand_gists → rerank → expand_doc_neighbors
                                                         ↓
                                                  build_context
                                                         ↓
                                                  validate_evidence
                                                    ↙            ↘
                                          mock_answer          insufficient_evidence
                                                    ↘            ↙
                                                      trace → END
```

### Node Details

**1. `route`** — Routing

- Uses `SemanticRouter` v2 by default (`use_semantic_router=True`).
- Produces `RoutePlan`, `SemanticRoutePlan`, and `EvidenceContract`.
- `semantic_route_trace()` bounds routing metadata for observability.
- Supports `force_enabled_sources` merging via `_merge_force_enabled_sources()`:
  adds coordinator-sticky sources not in the plan and enables disabled ones.

**2. `retrieve`** — Candidate Retrieval

- Iterates enabled sources from the `RoutePlan`.
- Dispatches to individual retrievers via `RetrieverDispatcher`.
- Applies `scoped_source_plan()` for chat-scoped document resolution.
- Caps results at `LANGGRAPH_MAX_DIRECT_RETRIEVAL_CANDIDATES` (160).
- Errors per-source collected, not fatal.

**3. `expand_gists`** — Gist Expansion

- Runs `dispatcher.gist_expander.expand()` on retrieved candidates.
- Expands gist-orientation candidates into bounded raw message spans.
- Caps expanded candidates at `LANGGRAPH_MAX_GIST_EXPANSION_CANDIDATES` (80).
- Query passed for relevance-aware expansion.

**4. `rerank`** — Reranking

- Merges base and expanded candidates into one pool.
- Uses `MemoryReranker.rank_with_trace()`.
- Applies `ranking_profile` from the `RoutePlan`.
- Query for reranking from `retrieval_query_for_reranking()`.

**5. `expand_doc_neighbors`** — Document Neighbor Expansion

- Expands document candidate chunks with adjacent chunks via `dispatcher.expand_document_neighbors()`.
- Replaces `reranked_candidates` with neighbor-augmented set.
- Errors caught non-fatally.

**6. `build_context`** — Context Construction

- Delegates to `ContextManagerAgent.build_context_packet()`.
- Produces the `ContextPacket` with budget allocations and selected evidence.
- Records `source_budgets` and per-source token usage.

**7. `validate_evidence`** — Evidence Contract Validation

- Checks the `EvidenceContract` against sources present in the `ContextPacket`.
- Four failure conditions via `evidence_failure_reason()`:
  - Raw span required but absent from packet.
  - Document citation required but absent.
  - Structured memory required but absent.
  - Only gist sources present when `must_not_answer_from_gist_only` is set.
- Conditional branch via `_answer_branch()`: `"answer"` or `"insufficient"`.

**8. `mock_answer` / `insufficient_evidence`** — Terminal

- `mock_answer`: produces `"MOCK ANSWER: evidence contract satisfied with sources X."`.
- `insufficient_evidence`: produces `"MOCK INSUFFICIENT EVIDENCE: {reason}."`.
- Neither node calls an LLM or generates real answers.

**9. `trace`** — Observability

- Collects route sources, candidate identities, source counts, dropped
  candidates, budgets, token usage, provenance validity, contract state,
  errors, and node-level timings.
- Candidate traces bounded: max 20, snippets ≤160 chars.
- Query text capped at 500 chars.

### Environment Variables

| Variable | Default | Purpose |
|----------|---------|---------|
| `LANGGRAPH_MAX_DIRECT_RETRIEVAL_CANDIDATES` | `160` | Cap on candidates from direct retrieval |
| `LANGGRAPH_MAX_GIST_EXPANSION_CANDIDATES` | `80` | Cap on gist-expanded candidates |

### Force-Enabled Sources

The `_merge_force_enabled_sources()` function accepts a `RoutePlan` and
`frozenset[str]` of source names to force-enable. It:

1. Enables any `SourcePlan` already in the plan whose `source` is in the set.
2. Appends new `SourcePlan` entries for sources not yet in the plan.
3. Skips the merge if no changes are needed (identity check via `changed` flag).
4. Sets `requires_retrieval=True` when sources are added.

This supports coordinator-sticky scopes (e.g., document QA forcing
`document_memory`) that the semantic router may not include by default.

### Routing Trace

`semantic_route_trace()` produces bounded routing metadata:

```python
{
    "routing_mode": "semantic_v2",
    "original_query": str,
    "normalized_query": str,
    "retrieval_query": str | original,
    "query_rewrite_applied": bool,
    "query_rewrite_reason": str,
    "language": str,
    "intents": [{intent, confidence, reason}],
    "temporal_scope": str,
    "retrieval_need": str,
    "memory_scope": str,
    "primary_scope": str,
    "required_scopes": [...],
    "task_context": str,
    "active_sources": [...],
    "retrieval_queries": [{purpose, allowed_sources, is_generated}],
    "confidence": float,
    "evidence_contract": {...},
}
```

### Key Differences from Native

| Aspect | Native | LangGraph |
|--------|--------|-----------|
| Router | `RoutingAgent` (rule/LLM/hybrid) | `SemanticRouter` v2 by default |
| Gist expansion | Implicit in retrieval | Explicit `expand_gists` node |
| Doc neighbor expansion | Not separated | Explicit `expand_doc_neighbors` node |
| Evidence validation | In context builder | Explicit `validate_evidence` node with contract |
| Answer generation | `ChatAgent` (real LLM) | Mock strings only |
| Database writes | Via `ShortTermMemoryAgent` | None |

## Write-Safety: Graph Is Read-Only

The LangGraph graph **never writes to the database**. It has no access to
`ShortTermMemoryAgent`, `LangMemStructuredMemoryState`, `LongTermStore`, or
any write path. It only reads through retrievers and produces a
`ContextPacket`.

All memory updates, message persistence, and gist generation happen through
the Native path — which always runs first, even in `LANGGRAPH_DEMO` mode.

The graph is intentionally incomplete for production:

- No real LLM answer generation.
- No database writes.
- No chat lifecycle actions (end, fork).
- No document ingestion.

---

## Dual-Path Safety: Native Always Runs First

In `coordinator_agent.py`, the orchestration sequence is:

1. **Native preparation always runs first.** The `CoordinatorAgent` builds
   the Native `OrchestrationResult` unconditionally.
2. If mode is not `NATIVE`, the LangGraph graph is invoked.
3. In `LANGGRAPH_DEMO` mode, the LangGraph result replaces Native as
   authoritative. If the graph fails, Native is the fallback.
4. Legacy context building (ShortTermMemoryAgent) still runs against Native.
5. The `ChatAgent` still uses Native infrastructure for answer generation.

This guarantees that even if the LangGraph graph is broken, the system
degrades gracefully to the Native answer.

## Orchestration Result and Comparison

### `OrchestrationResult`

Returned by `run_read_only_langgraph_orchestration()`. Contains:

- `context_packet`: the authoritative `ContextPacket`
- `trace`: `WorkflowTrace` with all retrieval, reranking, and context metadata
- `mode`: `LANGGRAPH_DEMO`
- `error`: concatenated errors string (or `None`)

The function accepts `force_enabled_sources` as a `frozenset[str]` — passed
through to the graph state for coordinator-sticky source enforcement.

### `OrchestrationComparison`

The `OrchestrationComparison` frozen dataclass captures bounded differences
between Native and LangGraph results without exposing content:

| Field | Meaning |
|-------|---------|
| `native_sources` | Source types in Native `ContextPacket` |
| `langgraph_sources` | Source types in LangGraph `ContextPacket` |
| `native_only_sources` | Sources only in Native |
| `langgraph_only_sources` | Sources only in LangGraph |
| `selected_candidate_overlap` | Count of shared (source, record_id) pairs |
| `native_selected_count` | Candidate count in Native |
| `langgraph_selected_count` | Candidate count in LangGraph |
| `native_prompt_tokens` | Estimated prompt tokens in Native |
| `langgraph_prompt_tokens` | Estimated prompt tokens in LangGraph |
| `token_difference` | LangGraph tokens minus Native tokens |

Used in `LANGGRAPH_SHADOW` mode for diagnostics. Never exposes candidate text
or transcript content.
