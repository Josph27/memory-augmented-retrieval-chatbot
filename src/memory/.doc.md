# src/memory/ — Typed Memory System

**TL;DR:** Three-layer memory architecture. Short-term loads recent messages from SQLite. Structured (LangMem-backed) extracts durable facts, preferences, and decisions into SQLite `long_term_memories`, synced to a **sqlite-vec** vector index for semantic search. Episodic (gists) compacts older messages into lossy orientation summaries — always expandable to raw spans. TTL system auto-expires transient memories. Dual-storage: SQLite is source of truth, sqlite-vec is a derived index.

---

## Architecture overview

```
Answer turn
  ShortTermMemory.build_context()
    → loads memory_state (SQLite)
    → loads recent messages (SQLite)
    → builds ShortTermContext
  ShortTermMemory.build_model_messages()
    → [system, system(memory), user, assistant, …]
  After answer:
    ShortTermMemory.update_memory_if_needed()
      → selects eligible ConversationUnit batch
      → LangMem extraction (LLM)
      → validates/normalizes (defense-in-depth)
      → SQLiteLongTermMemoryStore.upsert()
        → StructuredMemoryVectorSync.sync_record()
          → LongTermMemoryVectorIndex.upsert_record() → sqlite-vec
  At chat-end:
    ShortTermMemory.process_all_for_chat_end()
    PreviousChatGistGenerator.finalize_chat()
```

### Three memory layers

| Layer | Source | Storage | Retrieval | Purpose |
|---|---|---|---|---|
| **Short-term** | Recent same-chat messages | SQLite `messages` | `recent_messages()` | Immediate conversation context |
| **Structured** | LangMem extraction from user messages | SQLite `long_term_memories` + **sqlite-vec** `vec_memories` table | `StructuredMemoryRetriever` (SQLite/vector/hybrid) | Durable cross-chat facts, preferences, decisions — 10 categories with TTL decay |
| **Episodic (gists)** | LLM or deterministic summarization | SQLite `chat_gists` | `CurrentChatGistRetriever`, `PreviousChatGistRetriever` | Lossy orientation, expandable to raw spans |

## Short-term memory (`short_term.py`, `constants.py`)

### Core concepts

**`ShortTermMemory`** is the memory hub. It serves two roles during a user turn:

1. **Context building** — loads structured memory state + recent raw messages into `ShortTermContext`
2. **Memory updating** — schedules and applies batched LangMem extraction from pending messages

### Context building

`build_context()` returns a `ShortTermContext` containing `memory_state` (SQLite JSON from `chat_memory_state`) plus `raw_messages` from `recent_messages_before_id()`.

`build_model_messages()` converts `ShortTermContext` into a chat-completions message list:

```
[system: main system prompt]          ← role="system"
[system: "Current structured memory:  ← role="system" (MEMORY_CONTEXT_ROLE)
  - user_facts.name: Alice
  - user_preferences.language: TypeScript"]
[user: "What was my name?"]          ← message.role preserved
[assistant: "You said your name is…"] ← message.role preserved
```

**Why two system messages?** Qwen and some models do not support `role="memory"`. Injecting structured memory as a second `system`-role message ensures compatibility.

### ConversationUnit batching

Messages are grouped into **`ConversationUnit`** objects — consecutive `user → assistant` pairs. Standalone messages form single-message units. **Pairs are never split** across batch boundaries.

Batch selection:

1. **Group** pending (non-summarized) messages into `ConversationUnit` objects
2. **Protect** the most recent N tokens (`MEMORY_RECENT_PROTECTION_TOKENS`, default 1,500)
3. **Trigger** — if eligible tokens ≥ trigger threshold, proceed
4. **Select** — take oldest fitting units up to `max_input_tokens` and `max_messages`
5. **Require** — batch must contain at least one user message

### Two scheduling profiles

| Profile | Trigger | Max input | Max messages | Used for |
|---|---|---|---|---|
| `online` | `MEMORY_UPDATE_TRIGGER_TOKENS` (1,000) | `MEMORY_UPDATE_MAX_INPUT_TOKENS` (4,000) | `MEMORY_UPDATE_MAX_MESSAGES` (64) | Active chat |
| `offline_replay` | `MEMORY_REPLAY_TRIGGER_TOKENS` (4,000) | `MEMORY_REPLAY_MAX_INPUT_TOKENS` (8,000) | See note below | Chat-end replay |

**`MEMORY_REPLAY_MAX_MESSAGES` discrepancy:** `constants.py` defines `MEMORY_REPLAY_MAX_MESSAGES = 2`, but `settings.py` pushes `MEMORY_REPLAY_MAX_MESSAGES = 128` into the environment. The `ShortTermMemory` constructor accepts `memory_replay_max_messages` as a parameter and falls back to `MEMORY_UPDATE_BATCH_SIZE` (6) when `None`. The effective runtime value depends on how the constructor is called — if the caller reads the env var, it gets 128; if it passes `None`, it gets 6.

### Three memory update policies

| Policy | Behavior |
|---|---|
| `scheduled` **(default in settings.py)** | Token-threshold-triggered batched processing. Uses the `online` profile. |
| `agentic_each_turn` | Fires LangMem after **every** user turn on the latest incomplete `ConversationUnit`. No token thresholds. Graceful no-ops consumed silently. |
| `chat_end_only` | All memory extraction deferred to chat-end. `update_memory_if_needed()` is a no-op during chat. |

The default in `settings.py` is `"scheduled"`. The `normalize_memory_update_policy()` function normalizes to this.

## Structured memory (`langmem_structured.py`, `structured_state.py`, `long_term_store.py`)

### Memory categories — 10 categories

```python
MEMORY_CATEGORIES = (
    "past_events",       # Permanent — "I took the ML exam in August 2024"
    "user_experiences",  # Permanent — "I have 5 years of Python"
    "user_facts",        # 365d — "I live in Munich"
    "user_state",        # 90d — "I'm stressed about the exam"
    "user_preferences",  # 365d — "User prefers concise answers"
    "upcoming",          # 40d — "Exam later this month"
    "procedural",        # 365d — "To deploy: docker compose up"
    "corrections",       # 365d — "User corrected X to Y"
    "third_party_facts", # 365d — "User's friend bought red pants"
    "opinions",          # 90d — "User thinks Rust is better than C++"
)
```

### TTL system — system-assigned expiration

The model never sees datetimes. It categorizes; the system assigns `expires_at` post-extraction:

| Category | TTL | Behavior |
|---|---|---|
| `past_events` | Permanent | Only explicit supersession removes |
| `user_experiences` | Permanent | Accumulated skills don't expire |
| `user_facts` | 365 days | Slowly-changing biographical |
| `user_preferences` | 365 days | Interaction/tool/style preferences |
| `procedural` | 365 days | Reusable methods/instructions |
| `corrections` | 365 days | Audit trail of corrections |
| `third_party_facts` | 365 days | Same durability as user_facts |
| `user_state` | 90 days | Transient emotional/situational |
| `opinions` | 90 days | Transient views/judgments |
| `upcoming` | 40 days | Imminent future events |

`_compute_expires_at(category)` returns ISO-8601 timestamp or `""` for permanent categories. Expired records are filtered out by `_filter_unexpired()` in list/search operations.

### LangMem extraction pipeline

`LangMemStructuredMemoryState` is the **active extraction backend** (replacing the deprecated `StructuredMemoryState`).

**Extraction flow:**

```
User messages (batch)
  → messages_to_langchain_messages() (prepend [message_id=N])
  → existing_memories_for_langmem() (convert active records)
  → LangMem create_memory_manager (LLM call, Pydantic schema)
  → normalize_langmem_outputs() (validate, dedupe, source-check)
  → merge_memory_records() (merge with existing)
  → SQLiteLongTermMemoryStore.upsert()
```

The `LangMemStructuredMemory` Pydantic schema defines category, key, value, confidence (0.0–1.0), status (active/superseded/deleted), and source_message_ids.

### Connection guard integration

`LangMemStructuredMemoryState` checks `ConnectionGuard` before invoking LangMem. If `InferenceServerUnreachable`, returns `MemoryUpdateResult(accepted=False, rejection_reason="langmem_update_failed: ...")`. This prevents wasted LLM calls when the inference server is down.

### Validation pipeline — defense-in-depth

1. **Category check** — must be in `MEMORY_CATEGORIES`
2. **Key normalization** — snake_case, ≤80 chars
3. **Value cleaning** — whitespace stripped, `[message_id=N]` markers removed
4. **Transcript rejection** — values matching role patterns rejected (`looks_like_transcript_text`)
5. **Vagueness rejection** — "the user is discussing" etc. rejected (`is_vague_memory`)
6. **Confidence clamping** — normalized to [0.0, 1.0]
7. **Duplicate handling** — `upsert_normalized_record()` upserts by category/key

Rejected batches return `accepted=False` with a rejection reason. During online scheduling, rejections are logged but non-fatal. During chat-end, rejections that aren't safe no-ops (`langmem_no_valid_memories`, `no_user_messages`) raise `ChatEndMemoryProcessingError`.

### Dual storage architecture

```
┌──────────────────────┐     sync on every upsert     ┌──────────────────────┐
│  SQLite              │  ───────────────────────────→ │  sqlite-vec          │
│  long_term_memories  │   StructuredMemoryVectorSync  │  vec_memories        │
│  (source of truth)   │                               │  (derived index)     │
│                      │                               │                      │
│  namespace/key       │                               │  semantic search     │
│  lexical search      │                               │  embedding-based     │
│  CRUD ops            │                               │  rebuild from SQLite │
└──────────────────────┘                               └──────────────────────┘
```

**SQLite is the source of truth.** The vector index is a **sqlite-vec** derived index (vec0 virtual table, 384-dim cosine distance), rebuilt from SQLite on demand. `StructuredMemoryVectorSync.sync_record()` fires on every `SQLiteLongTermMemoryStore.upsert()` call — if SQLite succeeds but vector sync fails, a `StructuredMemoryVectorSyncError` is raised.

**Stable vector IDs:** `namespace::path::memory_id` (e.g., `memory::user_facts::user_facts:name`)

**Vector index internals:**

- Uses `BAAI/bge-small-en-v1.5` embeddings (384-dim)
- `LongTermMemoryVectorIndex` wraps the `vec_memories` vec0 table
- `index_records()` batch-indexes active records
- `search()` performs cosine similarity search returning `(rowid, similarity_score)`
- `_load_sqlite_vec()` creates the vec0 table on first use
- Supports a `VectorIndexBackend` protocol for test injection

### Namespace structure

| Namespace | Scope | Examples |
|---|---|---|
| `("memory", category)` | Category-scoped memories | `("memory", "user_facts")`, `("memory", "opinions")` |
| `("chat", chat_id, "structured_memory")` | Chat-scoped memories | `("chat", "abc123", "structured_memory")` |

`category_namespace(category)` returns `("memory", category)` — all categories are under `("memory", ...)`. The chat-scoped namespace is only added to `structured_memory_namespaces()` when `chat_id` is provided.

### Retrieval modes

`SQLiteLongTermMemoryStore` supports three retrieval modes:

| Mode | Backend | Good for |
|---|---|---|
| `sqlite` | Lexical search via `search()` | Exact key/category matches |
| `vector` | `LongTermMemoryVectorIndex.search()` via `StructuredMemoryRetriever` | Semantic similarity |
| `hybrid` **(default in settings.py)** | Both, via `StructuredMemoryRetriever` | Maximum recall |

The default `STRUCTURED_MEMORY_RETRIEVAL_MODE` in `settings.py` is `"hybrid"`.

### LongTermMemoryStore implementations

Two implementations:

- **`SQLiteLongTermMemoryStore`** — production store. CRUD on `long_term_memories` table, auto-syncs to vector index. Lazy-inits `StructuredMemoryVectorSync.from_env()` when mode is `vector` or `hybrid`.
- **`LangGraphInMemoryLongTermMemoryStore`** — `LangGraph InMemoryStore` wrapper for tests.

## Episodic memory (gists)

### Current-chat gists (`chat_gist_summarizer.py`)

`CurrentChatGistSummarizer` creates lossy orientation summaries from older same-chat messages. **Disabled by default** (`CURRENT_CHAT_GIST_GENERATION_ENABLED=False`).

When enabled, `process_current_chat_gist_batch()`:

- Takes oldest `GIST_MAX_MESSAGES_PER_GIST` (30) un-gisted messages
- Preserves newest `GIST_KEEP_RECENT_MESSAGES` (10) unsummarized
- Requires ≥ `GIST_MIN_MESSAGES_TO_SUMMARIZE` (20) messages to trigger
- Excludes latest user message from compression
- Writes to `chat_gists` table with `source_type="current_chat_gist"`

Gist output format:

```json
{
  "summary": "concise paragraph",
  "topics": ["short topic"],
  "decisions": ["decision made"],
  "open_tasks": ["unfinished task"],
  "important_facts": ["fact likely to matter later"],
  "corrections": ["correction made by the user"]
}
```

### Previous-chat gists (`previous_chat_gist.py`)

`PreviousChatGistGenerator` creates episodic summaries for finished chats.

- `PREVIOUS_CHAT_GIST_GENERATION_ENABLED` (default: `true`)
- `PREVIOUS_CHAT_GIST_EXTRACTOR` (default: `"llm"`; alt: `"deterministic"`)
- `PREVIOUS_CHAT_GIST_MAX_MESSAGES_PER_GIST` (default: 5, from `settings.py`)
- `FallbackChatGistExtractor` wraps primary extractor with deterministic fallback

At chat-end, `finalize_chat()` processes all pending messages in bounded batches:

- Assistant-only batches: valid no-ops (marked gist-processed without gist)
- Invalid extractor output: raises `PreviousChatGistFinalizationError`
- LLM failures stop iteration — remaining messages retried on next consolidate

`generate_for_existing_chats()` backfills gists for chats without one.

### Gists as orientation, not proof

Both gist types carry `evidence_role: "orientation_only"` in metadata. They are never used as proof. Gists are expandable to exact raw message spans via `GistRawSpanExpander` (in `src/retrieval/`).

## Memory trace (`memory_trace.py`)

`format_saved_memory_trace()` and `format_retrieved_memory_trace()` format memory operations for console output. Controlled by `DEMO_MEMORY_TRACE` env var.

Chainlit-compatible Markdown formatters:

- `format_saved_memories_markdown` — "🧠 Long-term memory saved"
- `format_retrieved_memories_markdown` — "🔎 Long-term memory retrieved"
- `format_retrieved_documents_markdown` — "📄 Document memory retrieved"

Utility converters: `memory_record_to_inspector_row()`, `memory_candidate_to_trace_row()`, `document_candidate_to_trace_row()`, `structured_memory_candidate_trace_rows()`.

Memory consolidation logs are written to `logs/memory/<chat_id>/batch_*.json` via `dump_memory_consolidation_log()`.

## Key design decisions

1. **SQLite is source of truth; sqlite-vec is derived.** Vector index uses `vec0` virtual table with 384-dim cosine distance. Sync is per-upsert, not eventually consistent. Rebuildable from SQLite.
2. **10 memory categories with TTL decay.** Permanent (past_events, user_experiences), long-lived (365d: facts, preferences, procedural, corrections, third-party), medium-lived (90d: state, opinions), short-lived (40d: upcoming). Expiration computed by system, not model.
3. **User/assistant pairs never split.** `ConversationUnit` batching preserves conversational coherence for LangMem.
4. **Memory injected as system message.** Uses a second `role="system"` instead of custom role for model compatibility.
5. **Online vs. offline profiles.** Different token budgets for live-chat (conservative) vs. chat-end (aggressive).
6. **Gists are orientation-only.** Marked `evidence_role: "orientation_only"` — never proof. Expandable to raw spans.
7. **LangMem is the active backend.** `StructuredMemoryState` is deprecated but retained for validator helpers.
8. **Validation is defense-in-depth.** Category check → key normalization → transcript rejection → vagueness rejection.
9. **Chat-end is fail-safe.** `process_all_for_chat_end()` raises on non-trivial failures to prevent silent memory loss. LLM failures stop iteration; messages remain pending for retry.
10. **Connection guard protects against inference outages.** `LangMemStructuredMemoryState` checks availability before invoking LangMem.
11. **Namespaces are category-scoped under `("memory", ...)`.** Not user/project/chat as old docs suggested. Chat-scoped namespace `("chat", chat_id, "structured_memory")` exists but is separate.
