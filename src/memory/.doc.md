# src/memory/ — Typed Memory System

**TL;DR:** Three-layer memory architecture. Short-term loads recent messages from SQLite. Structured (LangMem-backed) extracts durable facts, preferences, and decisions into SQLite, synced to a Chroma vector index for semantic search. Episodic (gists) compacts older messages into lossy orientation summaries — always expandable to raw spans. Dual-storage design: SQLite is source of truth, Chroma is a derived index.

---

## Architecture overview

```
┌─────────────────────────────────────────────────────────────────┐
│                        Answer turn                              │
│  ShortTermMemory.build_context()                                │
│    → loads memory_state (SQLite)                                │
│    → loads recent messages (SQLite)                             │
│    → builds ShortTermContext                                    │
│  ShortTermMemory.build_model_messages()                         │
│    → [system, system(memory), user, assistant, …]               │
│  After answer:                                                  │
│    ShortTermMemory.update_memory_if_needed()                    │
│      → selects eligible ConversationUnit batch                  │
│      → LangMem extraction (LLM)                                 │
│      → validates/normalizes                                     │
│      → SQLiteLongTermMemoryStore.upsert()                       │
│        → StructuredMemoryVectorSync.sync_record()               │
│          → LongTermMemoryVectorIndex.upsert_record() → Chroma   │
│  At chat-end:                                                   │
│    ShortTermMemory.process_all_for_chat_end()                   │
│    PreviousChatGistGenerator.finalize_chat()                    │
└─────────────────────────────────────────────────────────────────┘
```

### Three memory layers

| Layer | Source | Storage | Retrieval | Purpose |
|---|---|---|---|---|
| **Short-term** | Recent same-chat messages | SQLite `messages` table | `recent_messages()` | Immediate conversation context |
| **Structured** | LangMem extraction from user messages | SQLite `long_term_memories` + Chroma `long_term_memory` collection | `StructuredMemoryRetriever` (SQLite/vector/hybrid) | Durable cross-chat facts, preferences, decisions |
| **Episodic (gists)** | LLM summarization of older messages | SQLite `chat_gists` table | `CurrentChatGistRetriever`, `PreviousChatGistRetriever` | Lossy orientation, expandable to raw spans |

## Short-term memory (`short_term.py`, `constants.py`)

### Core concepts

**`ShortTermMemory`** is the memory hub. It serves two roles during a user turn:

1. **Context building** — loads structured memory state + recent raw messages into `ShortTermContext`
2. **Memory updating** — schedules and applies batched LangMem extraction from pending messages

### Context building

`build_context()` returns a `ShortTermContext` containing the current `memory_state` (SQLite JSON from `chat_memory_state`) plus the last N raw messages.

`build_model_messages()` converts `ShortTermContext` into a chat-completions message list:

```text
[system: main system prompt]          ← role="system"
[system: "Current structured memory:  ← role="system" (MEMORY_CONTEXT_ROLE)
  - user_facts.name: Alice
  - preferences.language: TypeScript"]
[user: "What was my name?"]          ← message.role preserved
[assistant: "You said your name is…"] ← message.role preserved
```

**Why two system messages?** Qwen 3.5 and some other models do not support a `role="memory"` message. Injecting structured memory as a second `system`-role message ensures compatibility without relying on non-standard roles.

### ConversationUnit batching

Messages are grouped into **`ConversationUnit`** objects — consecutive `user → assistant` pairs. A standalone message (e.g., a user message not followed by an assistant reply) forms its own single-message unit.

**User/assistant pairs are never split.** If a batch boundary would cut a `user → assistant` pair, the entire unit is either included or excluded. This ensures LangMem always sees complete conversational turns, preventing confusion from partial exchanges.

Batch selection works as follows:

1. **Group** pending (non-summarized) messages into `ConversationUnit` objects
2. **Protect** the most recent N tokens of conversation (controlled by `MEMORY_RECENT_PROTECTION_TOKENS`)
3. **Trigger** — if eligible tokens ≥ `MEMORY_UPDATE_TRIGGER_TOKENS`, proceed to batch selection
4. **Select** — take the oldest fitting units up to `max_input_tokens` and `max_messages`
5. **Require** — the batch must contain at least one user message (assistant-only batches are rejected)

### Two scheduling profiles

| Profile | Trigger | Max input | Max messages | Used for |
|---|---|---|---|---|
| `online` | `MEMORY_UPDATE_TRIGGER_TOKENS` (1,000) | `MEMORY_UPDATE_MAX_INPUT_TOKENS` (4,000) | `MEMORY_UPDATE_MAX_MESSAGES` (64) | During active chat |
| `offline_replay` | `MEMORY_REPLAY_TRIGGER_TOKENS` (4,000) | `MEMORY_REPLAY_MAX_INPUT_TOKENS` (8,000) | `MEMORY_REPLAY_MAX_MESSAGES` (2) | Chat-end replay |

During chat, `update_memory_if_needed()` fires after each answer if the trigger threshold is met. At chat-end, `process_all_for_chat_end()` flushes every pending message in batches regardless of thresholds.

### Three memory update policies

The `MEMORY_UPDATE_POLICY` env var selects when LangMem extraction fires:

| Policy | Behavior |
|---|---|
| `agentic_each_turn` **(default)** | Fires LangMem after **every** user turn on the latest incomplete `ConversationUnit`. No token thresholds. Graceful no-ops (LangMem finds nothing) are consumed silently — the batch is still marked processed. |
| `scheduled` | Token-threshold-triggered batched processing. Uses the `online` profile (1,000 token trigger, 4,000 max input, 64 max messages). |
| `chat_end_only` | All memory extraction deferred to chat-end. `update_memory_if_needed()` is a no-op during the chat. |

The `_select_latest_turn_batch()` method picks only the newest incomplete user-assistant pair for `agentic_each_turn`, bypassing all budgets, caps, and protection windows (`allow_noop=True`).

## Structured memory (`langmem_structured.py`, `structured_state.py`, `long_term_store.py`)

### LangMem extraction pipeline

`LangMemStructuredMemoryState` is the **active extraction backend** (replacing the deprecated `StructuredMemoryState` which used raw JSON operations).

**Extraction flow:**

```
User messages (batch)
  → messages_to_langchain_messages() (prepend [message_id=N])
  → existing_memories_for_langmem() (convert active records)
  → LangMem create_memory_manager (LLM call with Pydantic schema)
  → normalize_langmem_outputs() (validate, dedupe, source-check)
  → merge_memory_records() (merge with existing)
  → SQLiteLongTermMemoryStore.upsert()
```

**Eight memory categories:**
`user_facts`, `project_facts`, `decisions`, `corrections`, `open_tasks`, `preferences`, `constraints`, `procedural`

### Validation pipeline

Each LangMem output goes through a multi-stage validation:

1. **Category check** — must be in `MEMORY_CATEGORIES`
2. **Key normalization** — snake_case, ≤80 chars
3. **Value cleaning** — whitespace stripped, `[message_id=N]` markers removed
4. **Transcript rejection** — values matching `user:` or `assistant:` patterns are rejected
5. **Vagueness rejection** — phrases like "the user is discussing" are rejected
6. **Source support** — at least 50% of claim tokens must overlap with source message tokens (`text_supports_claim`)
7. **Confidence clamping** — normalized to [0.0, 1.0]

Rejected batches return `accepted=False` with a rejection reason. During online scheduling, rejections are logged but non-fatal. During chat-end processing, rejections that aren't safe no-ops (`langmem_no_valid_memories`, `no_user_messages`) raise `ChatEndMemoryProcessingError`.

### Dual storage architecture

```
┌──────────────────────┐     sync on every upsert     ┌──────────────────────┐
│  SQLite              │  ───────────────────────────→ │  Chroma              │
│  long_term_memories  │   StructuredMemoryVectorSync  │  long_term_memory    │
│  (source of truth)   │                               │  (derived index)     │
│                      │                               │                      │
│  namespace/key       │                               │  semantic search     │
│  lexical search      │                               │  embedding-based     │
│  CRUD ops            │                               │  rebuild from SQLite │
└──────────────────────┘                               └──────────────────────┘
```

**SQLite is always the source of truth.** Chroma is a derived index, rebuilt from SQLite on demand. `StructuredMemoryVectorSync.sync_record()` fires on every `SQLiteLongTermMemoryStore.upsert()` call — if SQLite succeeds but Chroma fails, a `StructuredMemoryVectorSyncError` is raised.

**Stable vector IDs:** `namespace::path::memory_id` (e.g., `user::default::semantic_memory::user_facts:name`)

### Namespace structure

| Namespace | Scope | Examples |
|---|---|---|
| `(user, default, semantic_memory)` | Global user facts, preferences, constraints, procedural, decisions, corrections | Name, language preference, tool choices |
| `(project, default, semantic_memory)` | Project-scoped facts, decisions, open tasks | Database choice, architecture decisions |
| `(chat, chat_id, structured_memory)` | Chat-scoped memories | Chat-specific constraints |

### Retrieval modes

`SQLiteLongTermMemoryStore` supports three retrieval modes controlled by `STRUCTURED_MEMORY_RETRIEVAL_MODE`:

| Mode | Backend | Good for |
|---|---|---|
| `sqlite` (default) | Lexical search via `search()` | Exact key/category matches |
| `vector` | `LongTermMemoryVectorIndex.search()` | Semantic similarity |
| `hybrid` | Both, via `StructuredMemoryRetriever` | Maximum recall |

## Episodic memory (gists)

### Current-chat gists (`chat_gist_summarizer.py`)

`CurrentChatGistSummarizer` creates lossy orientation summaries from older same-chat messages. **Disabled by default** (`CURRENT_CHAT_GIST_GENERATION_ENABLED`).

When enabled, `process_current_chat_gist_batch()`:

- Takes the oldest `GIST_MAX_MESSAGES_PER_GIST` (30) un-gisted messages
- Preserves the newest `GIST_KEEP_RECENT_MESSAGES` (10) unsummarized
- Requires ≥ `GIST_MIN_MESSAGES_TO_SUMMARIZE` (20) messages to trigger
- Excludes the latest user message from compression

Gist output format:

```json
{
  "summary": "concise paragraph",
  "topics": ["short topic"],
  "decisions": ["decision made"],
  "open_tasks": ["unfinished task"],
  "important_facts": ["fact likely to matter later"],
  "corrections": ["correction made by the user"]
}
```

### Previous-chat gists (`previous_chat_gist.py`)

`PreviousChatGistGenerator` creates episodic summaries for finished chats. Controlled by `PREVIOUS_CHAT_GIST_GENERATION_ENABLED`.

At chat-end, `finalize_chat()` processes all pending messages in bounded batches. Assistant-only batches are valid no-ops (marked gist-processed without generating a gist). Invalid extractor output is fatal — the chat end fails.

A `DeterministicPreviousChatGistExtractor` (no LLM) is available for tests and offline CLI use.

### Gists as orientation, not proof

Both gist types carry `evidence_role: "orientation_only"` in their metadata. They provide coarse conversational context but are never used as proof. When evidence is needed, gists are always expandable to exact raw message spans via `GistRawSpanExpander` (in `src/retrieval/`).

## Memory trace (`memory_trace.py`)

`format_saved_memory_trace()` and `format_retrieved_memory_trace()` format memory operations for console output. Controlled by `DEMO_MEMORY_TRACE` env var.

Chainlit-compatible Markdown formatters (`format_saved_memories_markdown`, `format_retrieved_memories_markdown`, `format_retrieved_documents_markdown`) are used for the debug UI.

## Key environment variables

| Variable | Default | Controls |
|---|---|---|
| `MEMORY_UPDATE_TRIGGER_TOKENS` | 1,000 | Online: tokens pending before triggering LangMem extraction |
| `MEMORY_UPDATE_MAX_INPUT_TOKENS` | 4,000 | Online: max batch token budget for LangMem |
| `MEMORY_RECENT_PROTECTION_TOKENS` | 1,500 | Online: tokens at the end of chat protected from summarization |
| `MEMORY_UPDATE_MAX_MESSAGES` | 64 | Online: max messages per batch |
| `MEMORY_REPLAY_TRIGGER_TOKENS` | 4,000 | Offline: tokens pending before triggering replay |
| `MEMORY_REPLAY_MAX_INPUT_TOKENS` | 8,000 | Offline: max batch token budget for replay |
| `MEMORY_REPLAY_MAX_MESSAGES` | 2 | Offline: max messages per replay batch |
| `CURRENT_CHAT_GIST_GENERATION_ENABLED` | `false` | Toggle current-chat gist compaction |
| `PREVIOUS_CHAT_GIST_GENERATION_ENABLED` | `true` | Toggle previous-chat gist generation |
| `STRUCTURED_MEMORY_RETRIEVAL_MODE` | `sqlite` | `sqlite`, `vector`, or `hybrid` |
| `DEMO_MEMORY_TRACE` | `false` | Print saved/retrieved memory trace blocks |
| `MEMORY_UPDATE_POLICY` | `agentic_each_turn` | `scheduled`, `agentic_each_turn`, or `chat_end_only`. Controls when LangMem extraction fires. |
| `MODEL_REQUEST_TIMEOUT` | 35 | Timeout in seconds for LangMem extraction calls |
| `LONG_TERM_MEMORY_COLLECTION` | `long_term_memory` | Chroma collection name for memory vectors |

## Key design decisions

1. **SQLite is source of truth, Chroma is derived.** Vector index can be rebuilt from SQLite at any time. Sync is best-effort per-upsert, not eventually consistent.
2. **User/assistant pairs never split.** `ConversationUnit` batching preserves conversational coherence for LangMem extraction.
3. **Memory injected as system message.** Uses a second `role="system"` message instead of a custom role for model compatibility.
4. **Online vs. offline profiles.** Different token budgets for live-chat extraction (conservative) vs. chat-end replay (aggressive).
5. **Gists are orientation-only.** Marked `evidence_role: "orientation_only"` — never treated as proof. Expandable to raw spans when needed.
6. **LangMem is the active backend.** The deprecated `StructuredMemoryState` (JSON-operation) still exists for validator helpers but is no longer the default.
7. **Validation is defense-in-depth.** Category checks → key normalization → transcript rejection → vagueness rejection → source support verification.
8. **Chat-end is fail-safe.** `process_all_for_chat_end()` raises on non-trivial failures to prevent silent memory loss.
