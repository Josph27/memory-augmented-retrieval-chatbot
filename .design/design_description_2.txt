[Glossary: lt mem = long  term memory, st mem = short term memory, vecdb = vecto database, doc = input documents for RAG, ]

Stores:
    vecdbs (separate dbs for easier manipulation)
        - documents (chroma db?)
            - meta = {src doc}
            - larger chunks (raw document chunks)
        - lt mem (sqlight?)
            - meta = {use_count, last used}
            - smaler chunks (condensed memory chunks)
    gists (sqlight indexed by chat id)  
        - retrieved lt mem list
        - st mem gist+text JSON {id, gist, info}
        - new memories JSON {id, memory, embed ID}


#DOC_INPUT_ACTION:
    -> @CHUNKING_AGENT chunk (Basic semanthic chunking)
    -> @EMBEDDING_AGENT embed chunks 

#CHAT_END_ACTION:
    #PROCESS_INTO_MEMORY_ACTION:
        1. Processing into new-memories list: 
        -> @MEMORY_UPDATE_AGENT 
            - retrieve relevant information compressions about the user from each not yet processed gist+text block.
            - try to assign compressions into existing new-memories
            - connect remaining compressions into new new-memories blocks for embedding. 
        2. Updating existing memories: 
            - foreach lt mem, @MEMORY_UPDATE_AGENT decides [update / delete / pass] based on gists+text.
        3. Commit changes into lt mem vecdb 
            - embed new new-memories 
            - for updated long term memories, remove old embedding by ID & re-embed.
    -> Mark chat as inactive in db.

#SINGLE_REQUEST_ACTION:
    -> query decomposition: 
        - @QUERY_AUGMENTATION_AGENT
            - decides on splitting into sub-queries
    -> semantic expansion:
        - @QUERY_AUGMENTATION_AGENT
            - injects keywords (prompt based, no dictionaries)
    ->  retrieval: (foreach sub-query), (foreach [lt mem, docs]):
        - @RETRIEVAL_AGENT 
            - evaluates whether doc vecdb retrieval is necessary for said sub-query.
            - lt mem (always) dense retrieval
            - docs (sometimes) dense retrieval
    ->  re-ranking: (foreach sub-query), (foreach [lt mem, docs]):
            - @RE_RANK_AGENT[CROSS_ENCODER]
                - scores chunks [0-1] based on query-chunk match
            - cutoff based on top k / score threshold, for both retrieval types. (mem_k > doc_k)    
    -> prompt stitching
        - static pre-defined placement to minimize lost in the middle: 
            {system prompt | doc_chunks (hi-lo score ordering), lt_mem_chunks (lo-hi ordering), user query (raw, no augmentation)}
    -> reply: 
        -> send to generator model API.
    -> reply processing:
        - output answer for user.
        - @GISTING_AGENT 
            - creates a gist from question/answer pair, appends into st mem.

#PRUNE_MEMORIES_ACTION:
    #PROCESS_INTO_MEMORY_ACTION (specified earlier)

#SUPPRESS_DOC_ACTION: 
    - fetch document's associated chunks (db -> doc -> chunks_ids)
    - set metadata for all chunks and the doc itself as "inactive", meaning they won't be able to participate in the queries.

#DELETE_MEMORY_ACTION:
    - memory id-> remove memory in lt mem store.

#ADD_MEMORY_ACTION:
    - inserts a new memory into the database with specified payload.

#NEW_CHAT_ACTION:
    - initiates an empty new chat in the database.

#CHAT_FORK_ACTION: 
    - duplicates all current chat information in the database, makes it an active chat.


User Interface: 
    (based on ChainLit)
    - TABS: 
        - landing_page: central new chat button, [list of active chats / "No active chat"] below.
        - chat_page: left sidebar: list of active chats, middle & right side: chat interface for communication with the system. Action  trigger buttons situated under the chat text input window.
        - chats_list_page: lislt of actice chats, followed by list of inactive chats, searchbar at the top. Clicking on active chats opens it in chat interface, hovering inactive chat offers a button to activate it again.
        - docs_list_page: central and only column: list of documents available to the system, inactive grayed out. Top button "add docs". each document has "suppress" and "delete" options triggering respective doc actions. 
        - memories_list_page: similar to docs list page, one central widget containing all lt memories in the system. Top button to "add memory" to forcefully add informatio nto the system, option to delete individual listed memories, search bar. 
    - Breadcrumbs menu: (always there, top panel)
        tabs:["Home" (= landing page), "Chats" (= chats list page), "Documents" (= docs list page), "Memories" (= memories list page)] 
    - ACTIONS ON TABS:
        - landing_page: NEW_CHAT_ACTION, 
        - chat_page: NEW_CHAT_ACTION, CHAT_END_ACTION, DOC_INPUT_ACTION, CHAT_FORK_ACTION
        - chats_list_page: 
        - docs_list_page: SUPPRESS_DOC_ACTION, DOC_INPUT_ACTION
        - memories_list_page: ADD_MEMORY_ACTION, DELETE_MEMORY_ACTION,
